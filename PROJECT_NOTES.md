# PROJECT NOTES

## 프로젝트 개요

IRIS(한국연구재단 범부처통합연구지원시스템), NTIS 등 정부과제 사업공고를 자동 수집·저장하고 로컬 웹에서 목록/상세/첨부파일을 열람하게 해주는 **개인용 로컬 에이전트**. 첨부파일(pdf/hwp/hwpx/zip)에 상세 정보가 담겨 있으므로 첨부 수집이 핵심이다. 현재는 **IRIS·NTIS 모두 완전 구현** 상태이며, 두 소스 모두 **접수예정·접수중·마감 3개 상태 전체**를 순차 수집한다.

## 아키텍처

- **Runtime**: Python 3.11~3.12. 의존성 선언은 `pyproject.toml`(setuptools), 실제 설치는 pip 또는 Docker 빌드 단계에서 수행.
- **레이어 구조** (`app/` 패키지):
  - `app/cli.py` — 스크래퍼 오케스트레이터 엔트리포인트(`python -m app.cli`). CLI argparse 없음. `sources.yaml`의 `scrape:` 섹션에서 실행 파라미터를 읽고 enabled/active 소스를 순서대로 실행. 우선순위: `scrape:` 섹션 > 소스별 설정 > 코드 default.
  - `app/config.py` — pydantic-settings 기반 `Settings` 싱글턴(`get_settings()`). `.env` 로더, 런타임 디렉터리 생성(`ensure_runtime_paths`). `base_url` 필드 제거됨 — 소스별 URL은 `sources.yaml`에서만 관리.
  - `app/sources/` — `constants.py`(소스 ID 상수), `config_schema.py`(`ScrapeRunConfig`/`SourceConfig`/`SourcesConfig` Pydantic 모델, `load_sources_config()`). `ScrapeRunConfig`는 `sources.yaml`의 `scrape:` 섹션 스키마(active_sources·max_pages·max_announcements·skip_detail·skip_attachments·dry_run·log_level). `load_sources_config()`는 `SOURCES_CONFIG_PATH` 환경변수 경로를 우선 사용한다(entrypoint 복사본 격리용).
  - `app/logging_setup.py` — loguru 루트 로거 초기화 + stdlib→loguru 브리지. `InterceptHandler`를 stdlib root logger에 부착해 uvicorn/starlette/FastAPI/SQLAlchemy/alembic 등 서드파티 로그도 동일 sink로 수렴. `configure_logging()`은 idempotent — `logger.remove()` 후 재설정하므로 중복 호출 안전. `_NOISY_LOGGER_NAMES`(sqlalchemy.engine 등)는 DEBUG 모드에서도 최소 INFO로 고정해 raw SQL 과잉 출력을 억제. loguru sink format은 KST 타임스탬프 명시(00040).
  - `app/timezone.py` — KST 헬퍼 모듈(00040). `KST = ZoneInfo("Asia/Seoul")`. `to_kst(dt)`: UTC tz-aware/naive(UTC 가정)/KST tz-aware 어떤 입력이든 KST tz-aware 정규화, None 통과. `now_kst()` / `now_utc()`: 저장은 now_utc, 표시는 now_kst. `format_kst(dt, fmt="%Y-%m-%d %H:%M")`: None 처리 포함. `kst_date_boundaries(date)`: KST 날짜의 [00:00, 24:00)을 UTC tz-aware 구간으로 반환 — Phase 5 GROUP BY 기반 예약. `_as_utc`(SQLite SELECT 보정)와 `to_kst`(KST 변환)는 별개 레이어이며 결합 사용(`_as_utc` → `to_kst` 순서).
  - `app/web/observability.py` — HTTP 요청/응답 로깅 미들웨어(`LoggingMiddleware`): 모든 요청에 `method path → status_code (Xms)` 형태로 처리 시간 기록. 전역 예외 핸들러(`add_exception_handlers`): 미처리 예외를 `logger.opt(exception=True).error()`로 강제 stack trace 출력 후 500 응답 반환 — uvicorn이 조용히 삼키는 500을 docker logs에 노출.

- **시각 처리**: DB의 모든 시간 컬럼은 timezone-aware UTC. 소스가 모호하면 timezone 추정 없이 UTC로 보존하고 원문 텍스트는 `raw_metadata` JSON에 함께 저장한다. 단, **SQLite 백엔드는 `DateTime(timezone=True)` 의 tz 정보를 실제로 저장하지 못해** SELECT 결과가 naive `datetime` 으로 돌아온다. tz-aware 시각(`datetime.now(tz=UTC)` 등)과 직접 비교하면 `TypeError`가 발생하므로, 비교 직전에 양쪽 값을 모두 UTC tz-aware 로 정규화하는 헬퍼(`_as_utc`)를 사용한다. 저장 컨벤션이 UTC라는 전제에 의존한다. **표시·로그·cron은 Asia/Seoul(KST) 단일 고정(00040)**: 저장은 `now_utc()`, 표시는 `to_kst()` + `format_kst()` 또는 Jinja2 필터(`kst_format`, `kst_date`)를 경유. `datetime.utcnow()` · naive `datetime.now()` 직접 사용 금지 — `app/timezone.py` 헬퍼로만 생성. 컨테이너 `TZ` env는 유지하되 코드는 `ZoneInfo("Asia/Seoul")`을 명시해 환경변수 비의존. 외부 응답(IRIS/NTIS) 날짜는 KST 가정 후 UTC 변환 저장.

## 주요 결정

- **목록·상세 수집은 httpx(HTTP-only), 첨부 수집만 Playwright**: IRIS 목록·상세 페이지는 SSR이라 httpx만으로 충분(탐사로 확인됨). 첨부 링크는 `javascript:` onclick으로만 노출되어 HTTP 직접 접근 불가 → Playwright headless로만 처리. 이 구분이 수집 단계 안정성을 높이고 의존성 무게를 줄인다.
- **Alembic 도입 완료 (00017)**: `alembic/versions/` 아래 migration 파일로 스키마를 관리. `init_db()` 는 Alembic Python API(`alembic.command.upgrade/stamp`)를 사용하여 stamp vs upgrade 분기를 자동 선택. 기존 DB는 `stamp head` 로 데이터 무변경, 신규 DB는 `upgrade head` 로 전체 스키마 생성. `env.py` 는 URL을 `alembic_config`에서 우선 읽어 호출자가 주입한 엔진을 존중한다.
- **SQLite 기본 + `DB_URL` 로 교체 가능**: 배포/공유 부담이 없는 단일 파일 DB가 개인용 목적에 맞음. Postgres 전환은 접속 문자열만 바꾸면 되도록 ORM 쪽은 JSON 범용 타입 사용.
- **첨부 바이너리는 DB 밖 FS에 저장**: hwp/zip이 수십 MB가 될 수 있어 BLOB 컬럼은 부적절. DB엔 경로/해시만. 중복/변경은 `sha256` 으로 판정.
- **페이지/공고 수집 상한을 sources.yaml·코드 기본값 2단계 우선순위로 관리**: 코드 기본값은 `max_pages=10, max_announcements=200`. 우선순위: `sources.yaml scrape:` 섹션 전역값 > 소스별 설정 > 코드 default. CLI 인자 단계는 00016에서 제거됨. 기본값을 보수적으로 유지해 차단 회피를 기본 정책으로 삼는 기조는 유지.
- **FastAPI 로컬 전용 + 세션 쿠키 인증(Phase 1b, 00021 에서 도입)**: "개인용 → 팀 공용" 로드맵에 따라 auth 를 도입했으나 전제는 여전히 로컬 네트워크 한정이다. **비로그인 열람은 유지** — 목록/상세 URL 은 기존 그대로이고, `current_user_optional` 로 None 이면 읽음 로직을 skip. 자유 회원가입(관리자 승인·이메일 검증 X), 비밀번호 bcrypt(passlib, rounds=12), 서버 사이드 세션(`UserSession` row) + HttpOnly/SameSite=Lax 쿠키(Secure=False, 로컬 HTTP 전제), 세션 ID `secrets.token_urlsafe(32)` 평문 DB 저장. JWT 는 의도적으로 선택하지 않았다(로컬 혁신성 대비 운영 복잡도 증가). 정책 상수와 근거는 `app/auth/constants.py` 에 주석과 함께 집중.
- **상태값을 한글 원문 그대로 Enum 값으로 보존**: IRIS 노출 텍스트와 1:1 매칭되어 필터/표시 변환 비용을 줄이기 위함.
- **증분 수집 전략 — DB 초기화 금지**: 스크래핑 재실행 시 기존 DB를 유지하고 변경된 공고만 갱신. `init_db()`는 Alembic stamp/upgrade로 스키마만 보장하고 기존 데이터를 절대 건드리지 않음. 이력 보존 및 불필요한 세부 수집 방지가 목적.
- **`received_at`을 변경 감지 비교 필드에서 제외**: 접수예정 상태에서 수집 시 공란이었다가 이후 보완되는 경우가 많아 잡음 트리거를 피하기 위해 의도적으로 제외. 핵심 비교 필드는 `title, status, deadline_at, agency` 4개로 고정.
- **2차 변경 감지 도입 (00019)**: 1차 감지(목록 UPSERT 시)는 sha256을 알 수 없어 첨부 변경 탐지 불가. 첨부 다운로드 완료 후 2차 감지를 추가해 첨부 개수·sha256 변동을 감지. `_upsert_announcement`를 비교 필드 파라미터화해 두 단계가 동일 함수를 재사용.
- **사용자 라벨링 데이터 보호가 변경 감지 리셋 정책의 핵심 근거**: 공고 내용이 실질 변경되면 `AnnouncementUserState(is_read)` 및 `RelevanceJudgment` 이력을 재설정 — 사용자 판단이 변경 전 내용 기준으로 오염되지 않도록. FavoriteEntry는 내용과 무관하게 유지.
- **FavoriteFolder depth 2 제약은 ORM validator**: SQLite ALTER TABLE 제약 추가 어려움과 단순 계층 구조 요구사항을 감안, DB CHECK 대신 ORM `@validates` 레벨에서 depth 검증.
- **즐겨찾기 항목은 announcement 단위(FavoriteEntry.announcement_id, task 00037 에서 전환)**: 00036 에서 채택했던 canonical 단위 저장 설계는 (a) 별을 누른 바로 그 공고가 아닌 다른 공고가 대표로 등록되는 문제와 (b) 동일 과제 여러 공고가 즐겨찾기 목록에 표시되지 않는 문제로 공식 폐기되었다. 이제 별은 공고(announcement) 하나마다 1건 저장되며, \"동일 과제 모두 저장\" 라디오는 같은 canonical 그룹의 is_current 공고 전체를 한 번에 등록하는 수단으로 기능한다. migration `c4a8d1e7b2f3` 이 기존 canonical 단위 row 를 대표 announcement(canonical_group 당 is_current=True + MIN id)로 이관하며, 대응 공고가 없는 row 는 삭제된다.
- **SQLite UNIQUE(parent_id IS NULL) 루트 폴더 동명 중복은 앱 레벨 SELECT 체크(00036)**: SQLite(및 Postgres)는 UNIQUE 제약에서 NULL을 "서로 다름"으로 취급하므로 parent_id=NULL 루트 폴더 동명 INSERT가 DB 제약으로 막히지 않는다. INSERT 전 SELECT로 직접 중복을 확인하는 app-level 체크 패턴을 적용.
- **Docker `scraper` 서비스를 `profiles: [scrape]` 로 격리**: 스크래퍼는 1회성 배치라 기본 `compose up` 에서 기동되면 안 됨. 명시적 `--profile scrape run --rm` 으로만 실행.
- **날짜 기반 수집 중단 조건 제거**: 정부 공고 사이트가 마감일을 `2200-01-01` 등 허위 값으로 기재하는 사례가 많아 날짜로 중단 여부를 판단하면 오작동 가능성이 큼. `list_scraper` 에서 날짜 기반 break 조건을 전부 제거하고 `max_pages` / `max_announcements` 상한만으로 수집을 제어한다.
- **외부 CDN 의존 없는 정적 리소스**: 오프라인/격리 환경에서도 UI가 그대로 동작하도록 CSS를 `app/web/static/` 패키지 내부에 둔다.
- **Canonical 스키마 — (B) 별도 테이블 선택**: `announcements`에 컬럼을 추가하는 (A)보다 `canonical_projects` 별도 테이블 + FK 참조(N:1) 구조(B)를 선택. 이유: cross-source 그룹 단위 검색·UI 표시 시 JOIN 없이 그룹 메타를 바로 조회할 수 있고, 나중에 그룹 수준 필드를 확장하기 쉽다.
- **IRIS 공식 식별자는 `ancmNo`**: 목록 API 전수 필드에서 비어 있는 레코드 0건. 1:1 고유 키가 아니라 "공고 묶음" 단위(하나의 공고번호에 여러 세부과제 ancmId가 대응). canonical_key 생성 전 NFKC 정규화 + 공백 제거 필수. 탐사 결과는 `docs/canonical_identity_design.md §2` 참조.
- **표시 timezone은 Asia/Seoul 단일 고정, 사용자별 설정 없음 (00040)**: 팀 공용 로컬 단일 운영 전제. Phase 6 사용자 설정 도입 전까지 Asia/Seoul 고정. `app/timezone.py` KST 헬퍼와 Jinja2 필터(`kst_format`, `kst_date`)가 이를 강제한다. `datetime.utcnow()` 등 naive 생성은 프로젝트 금지 패턴.
- **APScheduler BackgroundScheduler(timezone=ZoneInfo("Asia/Seoul")) 설정 (00040)**: cron 표현식이 KST 기준으로 해석됨. `30 9 * * *` = KST 09:30. timezone 변경 후 jobstore 기존 잡의 next_run_time이 재해석되어 일회성 점검 필요.
- **외부 응답(IRIS/NTIS) 날짜는 KST 가정 → UTC 변환 저장 (00040)**: 소스가 timezone을 명시하지 않지만 KST 기준으로 게시된다는 관찰 기반. 원문은 `raw_metadata`에 보존해 가정이 틀렸을 때 backfill 가능(`scripts/backfill_kst_assumption.py`).
- **`compute_canonical_key` 공식 키 우선 + fuzzy fallback**: official 키(ancmNo + 공고명 합성, 00039 도입) 를 우선 사용하고, 없으면 (제목 불용어 제거 + 주관기관 정규화 + 마감연도) 조합. NTIS cross-source 실매칭에서도 동일 로직 재사용 확인(fuzzy 매칭 → official 승급 경로 검증됨). fuzzy 임계값·튜닝 근거 + 신 키 포맷·정규화 규칙은 `docs/canonical_identity_design.md §4-1 / §11-6` 참조.
- **official canonical_key 합성 키 도입 (00039, 2026-04-28)**: 공고번호 단독 official 키에서 동일 ancmNo 아래 세부 공고가 다른 케이스(false-positive)가 발생해 포맷을 `"official:{normalized_ancmNo}::{normalized_title}"` 로 변경. title 정규화는 NFKC → NTIS suffix 제거(`\s*_\([0-9]{4}\).*$`) → 공백 전체 제거. 분석 §8-4 예측(canonical_projects 52→56, 다중 그룹 7→5, 최대 그룹 크기 4→2) 이 실데이터에서 그대로 재현됨 — false-positive 0건/false-negative 0건. 구분자 `::` 는 ancmNo 가 단일 콜론을 포함하지 않는다는 가정 위에서 사용. fuzzy scheme 은 변경 없음. DB 가 초기화된 상태에서 신규 수집되어 backfill 불필요. 분석·검증: `docs/duplicate_detection_analysis.md` + `docs/00039-3-verification.md` + `docs/canonical_identity_design.md §11-6`.
- **NTIS는 httpx 단독 수집 (Playwright 불필요)**: 목록·상세·첨부 모두 httpx POST SSR로 처리. IRIS와 달리 AJAX API 없음. 첨부도 `wfUid + roTextUid` 파라미터로 httpx POST 직접 다운로드 가능(JS onclick 우회 불필요). `AttachmentDownloader`는 소스별 링크 추출 분기를 adapter 계층에서 처리하고 downloader 자체는 소스를 모름.
- **sources.yaml에 credentials 슬롯(optional)**: 로그인이 필요한 소스를 위한 `credentials.username`/`credentials.password` 필드를 `SourceConfig`에 예약. NTIS는 게스트 수집이 가능하므로 현재 미사용. 실제 값은 `.env`로만 주입하고 yaml에는 키 이름만 남긴다.
- **CSRF 토큰 대신 same-origin 체크(Phase 1b)**: 로컬 팀 전용 전제라서 CSRF 토큰 기반 방어는 과잉. POST 인증 라우트는 `ensure_same_origin` 의존성으로 Origin/Referer 헤더만 netloc 비교한다. 헤더가 없는 비-브라우저 호출(curl 등)은 통과시켜 스크립트 운영 마찰을 줄인다.
- **읽음/안읽음 UI 는 로그인 시만 활성(Phase 1b)**: `AnnouncementUserState.is_read` 조회를 위해 현재 페이지 `announcement_id` 집합을 한 번에 SELECT(`get_read_announcement_id_set`) 해 N+1 을 제거. 상세 진입 시 `mark_announcement_read` 로 UPSERT. 비로그인은 전체 skip — 기본 경로에 오버헤드 0. IRIS/NTIS 는 announcement 단위로 독립 관리(canonical 묶음에 종속시키지 않음) — 같은 공고의 소스별 버전을 각각 읽음 처리 가능하도록.
- **첫 관리자 계정은 CLI 로만 생성**: 자유 회원가입은 항상 `is_admin=False`. 관리자 UI 는 Phase 2+ 이므로, 그때까지는 `scripts/create_admin.py` (getpass 2회 + argparse username/--email) 를 통해서만 관리자를 생성한다. 컨테이너 내부 TTY 실행을 전제로 설계.
- **sources.yaml을 gitignore 전환 + template 도입 (00022)**: sources.yaml이 git에 커밋되어 있으면 브랜치 전환 시 로컬 수정본이 롤백되는 문제가 있다. sources.yaml을 `.gitignore`에 추가하고, `sources.yaml.template`(플레이스홀더 포함)만 버전 관리. 이미지에도 template을 COPY해 두어 마운트 누락 시 폴백으로 사용.
- **Dockerfile에 alembic 산출물 포함 필수 (00022)**: `alembic.ini`와 `alembic/` 디렉터리가 이미지에 없으면 `entrypoint.sh`의 `alembic upgrade head`가 "No 'script_location' key found" 오류로 실패한다. 이미지 빌드 시 반드시 COPY해야 하며, migration 버전 추가 시 이미지 재빌드 필요.
- **웹 수집 제어는 docker-in-docker 대신 호스트 docker.sock 원격 조작 (00025)**: `dind` 방식은 이미지·권한·볼륨 재설계 부담이 커서, app 컨테이너에 호스트 `/var/run/docker.sock` 을 마운트하고 docker CLI + compose v2 플러그인을 설치해 `docker compose` 를 호출하는 방식을 선택. 호스트 dockerd 가 실제 컨테이너를 관리하므로 scraper 의 `./data`·`./app`·`./sources.yaml` 바인드가 기존 CLI 경로와 동일하게 적용된다. 대가: docker.sock 마운트는 사실상 호스트 root 권한 부여와 동등 — FastAPI 외부 노출 금지 전제가 이 구성에서 더 강하게 요구된다. 설계 근거는 `docs/scrape_control_design.md §5`.
- **compose v2 필수 (docker-compose v1 금지, 00025)**: v2 가 SIGTERM 을 관리 컨테이너 PID 1 로 공식 전파한다. v1 `docker-compose` 는 전파 동작이 검증되지 않아 "중단 → 공고 마무리 후 cancelled" 흐름이 깨질 수 있다. Dockerfile 이 compose v2 바이너리를 `/usr/libexec/docker/cli-plugins` 에 직접 배치해 `docker compose` 서브커맨드 형태로만 호출한다.
- **subprocess 는 새 프로세스 그룹 + 파일 redirect (00025)**: 웹/스케줄러 → scraper subprocess 기동 시 `start_new_session=True` (`os.setsid`) 로 새 프로세스 그룹 생성. 중단은 `os.killpg(pgid, SIGTERM)` 로 그룹 전체 전파 — compose v2 가 컨테이너 PID 1 로 릴레이. detach/데몬화 금지(부모-자식 신호 체인이 끊김). stdout/stderr 는 PIPE 가 아닌 `data/logs/scrape_runs/{run_id}.log` **파일 redirect** — PIPE 버퍼가 가득 차면 자식이 write 에서 블로킹된다. Windows 호환은 포기(`os.setsid` POSIX 전용).
- **ScrapeRun 을 세 경로 공통 lock 으로 사용 (00025)**: 웹(trigger='manual')/스케줄러(trigger='scheduled')/CLI(trigger='cli') 모두 `create_scrape_run` 이 단일 트랜잭션 안에서 "running row 존재 검사 → INSERT" 를 수행. running row 가 곧 "실행 중 flag" — 별도 파일 락·메모리 뮤텍스를 도입하지 않는다. 웹 startup 에서 `cleanup_stale_running_runs()` 가 pid 없는/프로세스 사라진 running row 를 failed 로 정리해 lock 자동 해제. APScheduler `start_scheduler()` 는 cleanup 이후에 호출해 race 를 피한다.
- **`SCRAPE_RUN_ID` 환경변수로 웹→CLI subprocess ScrapeRun 소유권 이전 (00034)**: 웹/스케줄러의 `start_scrape_run` 은 `create_scrape_run` 으로 running row 를 먼저 INSERT 한 뒤 subprocess 를 기동한다. 기존 CLI 가 그대로 "running row 존재 검사" 를 수행하면 방금 생성된 자기 row 를 '이미 진행 중인 다른 수집' 으로 오판해 exit(2) 하는 자기참조 버그가 있었다. 해결: `SCRAPE_RUN_ID` 환경변수로 id 를 주입하면 CLI 는 새 row 를 만들지 않고 해당 row 를 이어받아 finalize 까지 수행한다. 미주입이거나 파싱 실패 시 기존 CLI 직접 실행 경로(trigger='cli' 로 자체 create_scrape_run) 로 동작해 하위 호환이 유지된다.
- **SIGTERM 처리는 atomic 경계 보존 (00025)**: CLI 가 SIGTERM 을 받으면 handler 는 모듈 레벨 `_cancel_flag` 만 세우고, 실제 break 는 **공고 루프/소스 루프의 시작 지점에서만** 수행. 현재 공고(첨부 다운로드 포함)는 끝까지 마무리 — Phase 1a 에서 확립한 "UPSERT + 변경 감지 + 사용자 라벨링 리셋은 같은 트랜잭션" 원칙을 중단 경로에서도 깨지 않기 위함. 종료 status 는 `cancelled`.
- **APScheduler 3.x 를 웹 프로세스 내부 BackgroundScheduler 로 (00025-6)**: 별도 worker 프로세스를 띄우지 않고 FastAPI 와 같은 프로세스 안에서 `BackgroundScheduler` 를 가동 — 운영 단순성이 팀 공용 로컬 배포 목적에 부합. JobStore 는 `SQLAlchemyJobStore` 단일(`scheduler_jobs` 테이블, APScheduler 가 자체 생성해 Alembic 관리 밖). 재시작 시 잡 자동 복원. `misfire_grace_time=300`·`coalesce=True`·`max_instances=1` 로 놓친 잡 합치기 및 중복 기동 방지. 4.x 는 API breaking change 라 `apscheduler>=3.10,<4.0` 으로 고정.
- **SCRAPE_ACTIVE_SOURCES 환경변수로 per-run 소스 override (00025)**: 웹 UI "지금 시작(source 선택)" 은 `sources.yaml` 파일을 in-place 수정하면 entrypoint 의 per-run 복사본 격리가 깨져 동시 실행 시 설정 경합이 난다. 파일 대신 `SCRAPE_ACTIVE_SOURCES` 환경변수(쉼표 연결 문자열, 예 `IRIS,NTIS`)로 주입 — `app/cli.py` 가 로드 후 우선 적용, 미설정/빈 값이면 `sources.yaml` 의 `active_sources` 를 그대로 사용.
- **컨테이너를 호스트 UID/GID 로 실행 (00028)**: `docker-compose.yml` 두 서비스에 `user: "${HOST_UID}:${HOST_GID}"` 적용. `.env` 에 명시 설정 필수(fallback 없음). 목적: `./data/` 하위 모든 런타임 생성 파일이 호스트 유저 소유가 되어 `sudo` 없이 관리 가능. 기존 root-owned 파일은 `sudo chown -R "$(id -u):$(id -g)" ./data/` 로 일회성 복구.
- **sources.yaml 웹 편집기는 원자적 쓰기 + 백업 필수 (00025-5)**: 편집 흐름은 "YAML syntax → `SourcesConfig.model_validate` 통과 시에만 저장". 저장 전 `data/backups/sources/YYYYMMDD_HHMMSS.yaml` 로 타임스탬프 백업. 쓰기는 `NamedTemporaryFile` + `os.replace` 원자적; 두 경로에서 `_overwrite_with_fsync` fallback: (a) EXDEV(bind mount 파일 단위, tmp 와 target 다른 device), (b) EACCES/EPERM(`/run/config/` 부모 dir 가 비쓰기 tmpfs 라 NamedTemporaryFile 생성 불가 — compose 표준 구성에서 발생하는 전형 케이스). 편집 대상은 bind mount 원본이라 현재 실행 중인 subprocess 의 per-run 복사본에는 영향 없음 — **다음** 수집부터 반영.
- **stdlib→loguru 브리지가 필수인 이유 (00030)**: uvicorn은 FastAPI 라우트에서 발생한 미처리 예외를 stdlib `logging`의 `uvicorn.error` 채널로 기록한다. loguru는 stdlib와 별도 시스템이라 브리지(`InterceptHandler`) 없이는 이 stack trace가 docker logs에 전혀 나타나지 않는다. LOG_LEVEL=DEBUG 설정에도 불구하고 로그가 비어 보이던 근본 원인이 이것이었다. 해결: `InterceptHandler`를 root logger에 부착 + uvicorn 로거 핸들러 명시적 교체.
- **APScheduler 백그라운드 컨텍스트에서 docker 바이너리를 절대경로로 탐색 (00029)**: `job_runner.py` 의 `scheduled_scrape` 가 호출하는 `build_compose_command` 는 `argv[0]` 을 문자열 `"docker"` 로 두면 APScheduler 실행 컨텍스트의 제한된 PATH 에서 `FileNotFoundError` 가 발생한다. `_resolve_docker_binary()` 를 도입해 `shutil.which` → 관행 경로(`/usr/bin/docker`, `/usr/local/bin/docker`) → 환경변수 `DOCKER_BINARY` 순으로 탐색하고 절대경로를 반환. 탐색 실패 시 `ComposeEnvironmentError` 로 ScrapeRun 을 failed 처리해 UI 에 진단 메시지를 노출한다.
- **`as_utc` 헬퍼를 auth 전용 → 공용으로 승격 (00025)**: `_as_utc` 로 `app/auth/service.py` 에만 있던 helper 를 `app/db/models.py` 로 이동해 `scrape_runs` 비교 로직에서 재사용. SQLite 의 `DateTime(timezone=True)` tz 손실은 프로젝트 공통 이슈라 컬럼 비교가 있는 모든 경로에서 동일 패턴이 필요하다.
- **관련성 단위는 canonical, 읽음 단위는 announcement — 혼용 금지 (00035)**: `RelevanceJudgment`의 unique 키는 `(canonical_project_id, user_id)` — 여러 소스의 announcement가 같은 canonical이면 판정을 공유한다. 반면 `AnnouncementUserState.is_read`는 `announcement_id` 단위 유지 — IRIS/NTIS 각 공고를 별도 읽음 처리 가능하도록. 이 두 단위는 의도적으로 다르며, 목록 row는 announcement이지만 관련성 배지는 canonical_id로 조회·표시한다.
- **관련성 판정 변경은 원자적 이관 (user_changed) — 덮어쓰기 금지 (00035)**: 기존 판정이 있을 때 새 판정을 저장하면 기존을 `RelevanceJudgmentHistory`로 이관(archive_reason="user_changed")한 뒤 신규 판정을 INSERT — 동일 트랜잭션. 직접 UPDATE로 덮으면 이력이 소실되므로 설계상 금지. content_changed 이관(공고 내용 변경 시)과 동일 패턴.
- **bulk 요청 두 가지 모드: ids vs filter (00035)**: 페이지 내 선택은 `ids` 모드(명시 목록), "현재 필터 결과 전체 M건 선택"은 `filter` 모드(서버가 필터 파라미터로 announcement id를 추출). `MAX_BULK_MARK` 상수로 한 번에 처리 가능한 상한을 제한. 클라이언트가 필터 재구성을 서버에 위임해 페이지를 넘는 대량 선택을 지원.
- **배지 클릭 vs row 클릭 충돌은 stopPropagation으로 해결 (00035)**: 목록 row 전체가 상세 링크(`<a>`)로 감싸져 있을 때 내부 배지 클릭이 상세 페이지로 이동하지 않도록 `event.stopPropagation()` + `preventDefault()` 패턴 적용. 이후 클릭 이벤트가 있는 인라인 위젯 추가 시 동일 패턴 사용.
- **수집 파이프라인 delta 기반 재설계 (00041)**: 수집 시 본 테이블 직접 UPSERT → ScrapeRun 종료 시 단일 트랜잭션(delta → 본 테이블 + snapshot + delta 비우기)으로 전환. 부분 실패 시 본 테이블·snapshot 변화 없음 보장이 목적. 첨부 파일 자체는 FS라 트랜잭션 밖 — 고아 파일은 GC 스크립트로 정리.
- **ScrapeSnapshot — KST 날짜 1 row UPSERT, payload 머지 (00041)**: `scrape_snapshots` 테이블에 `snapshot_date`(KST 날짜) UNIQUE 1 row. 같은 날 여러 ScrapeRun 결과는 `merge_snapshot_payload(existing, new)`로 합산. 5종 카테고리: new·content_changed·transitioned_to_{접수예정,접수중,마감}. 머지 규칙: new·content_changed는 ID set union; transitioned_to_X는 동일 id 마지막 to 갱신 + from==to면 제거(실질 변화 없음). payload는 JSON 범용 컬럼.
- **고아 첨부파일 GC 정책 (00041)**: 첨부 다운로드는 트랜잭션 보호 불가(FS). delta apply 실패/SIGTERM 시 다운로드된 파일이 DB 없이 남을 수 있음. `scripts/gc_orphan_attachments.py`(--dry-run 옵션)가 `data/downloads/` FS vs DB `attachments.file_path` 대조해 정리. APScheduler 일 1회 또는 수동 실행.
- **대시보드 캘린더는 순수 JS/CSS 자체 구현 선택 (00042)**: Pikaday 같은 외부 라이브러리 대신 월별 그리드 캘린더를 직접 구현. 이유: 외부 CDN 금지 컨벤션상 번들링이 필요한데, snapshot_date 강조·클릭 비활성 등 커스텀 요구사항이 많아 자체 구현이 번들 무게 대비 유리. 구현 결정은 `docs/dashboard_design.md`에 트레이드오프 포함.
- **snapshot 변화 0건 수집 날은 캘린더 비활성 — 디자인 의도 (00042)**: 5a의 `merge_snapshot_payload`는 변화 0건이면 `scrape_snapshots` row를 생성하지 않는다. 대시보드 캘린더는 `snapshot_date` 존재 여부로 가용성을 판정하므로, "수집은 했지만 변화 없음"인 날은 비활성으로 표시된다 — 의미 없는 기준일 선택을 막는 의도적 설계.
- **대시보드는 완전 read-only, 서버 사전 계산 + JSON 임베드 (00042)**: 대시보드 내 사용자 액션·POST 엔드포인트 없음. Chart.js 추이 차트 데이터는 서버에서 사전 계산 후 페이지에 JSON으로 임베드(별도 API 호출 없음). 컨트롤 변경(기준일/비교 대상) 시 즉시 페이지 reload — 자동 갱신 없음.
