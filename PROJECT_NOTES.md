# PROJECT NOTES

## 프로젝트 개요

IRIS(한국연구재단 범부처통합연구지원시스템), NTIS 등 정부과제 사업공고를 자동 수집·저장하고 로컬 웹에서 목록/상세/첨부파일을 열람하게 해주는 **개인용 로컬 에이전트**. 첨부파일(pdf/hwp/hwpx/zip)에 상세 정보가 담겨 있으므로 첨부 수집이 핵심이다. 현재는 **IRIS·NTIS 모두 완전 구현** 상태이며, 두 소스 모두 **접수예정·접수중·마감 3개 상태 전체**를 순차 수집한다.

- **Runtime**: Python 3.11~3.12. 의존성 선언은 `pyproject.toml`(setuptools), 실제 설치는 pip 또는 Docker 빌드 단계에서 수행.

- **데이터 흐름** (현재 활성):
  1. CLI가 `init_db()` (멱등, 기존 DB 재사용) → `load_sources_config()` → enabled 소스 목록 결정
  2. 각 소스마다 `get_adapter()` → `adapter.scrape_list()` → 목록 수집. 각 공고 `upsert_announcement(...)` → `UpsertResult` 반환.
  3. `UpsertResult.needs_detail_scraping` 기준으로 상세 수집 여부 결정: `unchanged` + 상세 이미 있으면 생략, 신규/변경/상태전이이면 항상 수집.
  4. 웹 UI는 목록 조회 + 공고 클릭 → 상세 페이지(`/announcements/{id}`) 표시. viewer 템플릿은 `source_type` 별로 분기.
  5. **[00008 추가]** 상세 수집 후 첨부파일 다운로드 단계 실행: `AttachmentDownloader`가 Playwright headless로 각 첨부 링크를 클릭 → 다운로드 완료 후 DB Attachment row 생성.
- **도메인 모델**: `Announcement` 1:N `Attachment`. 공고 UPSERT 키는 `(source_type, source_announcement_id)` 복합 UNIQUE. `is_current=True`인 row가 현재 버전. 상태는 `AnnouncementStatus` Enum(`접수중/접수예정/마감`, 한글 원문 값). Attachment는 `(announcement_id, original_filename)` 복합 UNIQUE + `sha256` 중복 체크.
- **Canonical 레이어**: `CanonicalProject` 테이블(schema B — 별도 테이블)이 `Announcement` 1:N을 FK로 묶는 그룹. `canonical_key` UNIQUE 컬럼(예: `official:과학기술정보통신부공고제2026-0455호`). 각 Announcement는 `canonical_project_id` FK로 참조. IRIS 공식 식별자는 `ancmNo`(공고번호 — 목록 API에 항상 존재). 탐사 결과는 `docs/canonical_identity_design.md` 참조.
- **Backfill/검증 스크립트**: `scripts/backfill_canonical.py`(기존 IRIS 데이터 일회성 canonical 재계산), `scripts/verify_canonical_iris.py`(IRIS 단독 중복 케이스 검증), `scripts/verify_canonical_cross_source.py`(IRIS↔NTIS cross-source 매칭 결과 샘플 출력 및 false positive/negative 검증).
- **증분 UPSERT 4-branch**: (a) 신규 → INSERT is_current=True. (b) 변경 없음 → no-op (상세 있으면 상세도 생략). (c) 상태 전이만 → in-place UPDATE. (d) 그 외 변경 → 기존 row 봉인(is_current=False) + 신규 row INSERT (이력 누적). 비교 필드: `title, status, deadline_at, agency` (`received_at` 제외 — 접수예정 시 공란 보완 많아 잡음 유발). Phase 1a에서 **첨부 비교** 추가: 1차 감지(목록 UPSERT 시, 기존 필드 기준) + 2차 감지(첨부 다운로드 완료 후, 첨부 개수·sha256 비교). `_upsert_announcement`를 "비교 필드 파라미터" 방식으로 재사용.
- **변경 시 리셋 (status 단독 전이 제외)**: 공고 내용 변경 감지 시 `AnnouncementUserState.is_read` 전부 False 재설정, `RelevanceJudgment`를 `RelevanceJudgmentHistory`로 이관(archive_reason="content_changed") 후 삭제, `FavoriteEntry` 유지. 전체 upsert와 동일 트랜잭션(atomic rollback 보장).
- **DB 백업**: `scripts/backup_db.py` — SQLite 파일 복사 + 타임스탬프. `data/backups/`에 최근 14개 rotation 보관. pg_dump 미지원. 사용법은 `README.USER.md` 참조.
- **상태 전이**: `docs/status_transition_todo.md` — IRIS 3개 상태 순차 수집 활성화됨(items 1·2 구현 완료). NTIS 등 신규 크롤러 구현 시 참고.
- **실행 환경**: Docker Compose. `app`(FastAPI UI)·`scraper`(1회성 배치, `profiles: [scrape]`)가 같은 이미지를 공유. 스크래퍼 실행 경로는 **3가지** — (1) 호스트 쉘에서 `docker compose --profile scrape run --rm scraper` (회귀 유지), (2) 웹 UI `/admin/scrape` 의 "지금 시작" 버튼 → app 컨테이너 내부 `subprocess` 가 호스트 dockerd 를 조작해 동일 명령 기동(Phase 2/00025), (3) APScheduler cron/interval 잡이 자동 기동. 호스트 Python 직접 실행은 지원 안 함.
  - **바인드 마운트**: `./data`, `./app`, `./sources.yaml:/run/config/sources.yaml`(app 서비스는 `:ro` 제거 — 편집기 쓰기 필요; scraper 서비스는 `:ro` 유지), `/var/run/docker.sock:/var/run/docker.sock`(**app 서비스만** — 웹이 subprocess 로 docker CLI 를 호출하기 위함), `./scripts:/app/scripts`(**app 서비스** — `create_admin.py` 등 운영자 CLI 스크립트를 `docker compose run --rm app python scripts/...` 로 실행하기 위함), `/etc/localtime:/etc/localtime:ro`(app·scraper 서비스 공통 — 컨테이너 시각이 호스트와 동일하게 동작하도록; `TZ=${HOST_TZ:-Asia/Seoul}` 환경변수와 짝을 이뤄 tzdata 미탑재 slim 이미지에서도 동작. DB 저장은 여전히 UTC, 이 설정은 loguru 로그 타임스탬프 등 프레젠테이션 계층에만 영향), `./.env:${HOST_PROJECT_DIR}/.env:ro`(**app 서비스** — 컨테이너 내부 compose subprocess 가 `--project-directory ${HOST_PROJECT_DIR}` 기준으로 `.env` 를 탐색하므로 호스트 `.env` 를 동일 경로에 읽기 전용으로 마운트).
  - **docker-in-docker 아님**: app 컨테이너는 자기 dockerd 를 띄우지 않고 호스트 `/var/run/docker.sock` 을 통해 호스트 dockerd 를 원격 조작한다. app 이미지에 `docker.io` + `docker-cli`(apt — 패키지 분리로 별도 명시 필요) + compose v2 플러그인(`docker-compose-linux-<arch>` 바이너리)을 설치하고 `docker-compose.yml` 자체를 이미지에 COPY 해 `docker compose -f /app/docker-compose.yml --project-directory $HOST_PROJECT_DIR ...` 로 호출한다. 상대경로 해석을 위해 `HOST_PROJECT_DIR`(호스트 프로젝트 루트 절대경로)를 `.env` 에 반드시 설정해야 한다 — 미설정 시 `ComposeEnvironmentError` 로 ScrapeRun 이 failed 마감.
  - **컨테이너 docker 그룹**: docker.sock 에 접근하려면 app 컨테이너 내부 유저가 호스트 `docker` 그룹 gid 를 보조 그룹으로 가져야 한다. `HOST_DOCKER_GID`(`.env`) → `docker-compose.yml` 의 `group_add` 로 전달. gid 불일치 시 "Permission denied on docker.sock".
  - **호스트 UID/GID 매핑 (00028)**: `app`·`scraper` 서비스 모두 `user: "${HOST_UID}:${HOST_GID}"` 로 실행. 컨테이너가 생성하는 모든 파일(SQLite·다운로드·로그·백업)의 owner 가 호스트 유저가 된다. `.env` 에 `HOST_UID=$(id -u)`, `HOST_GID=$(id -g)` 를 반드시 설정해야 하며, 자동 fallback 없음(미설정 시 기동 실패 또는 root 로 실행). `group_add(HOST_DOCKER_GID)` 와 독립적으로 동작한다.
  - **Playwright 브라우저 경로 공용화 (00028)**: `PLAYWRIGHT_BROWSERS_PATH=/opt/playwright` 로 `/root/.cache` 대신 공용 경로에 설치. 빌드 시 `chmod -R a+rX` 적용해 비특권 UID 기동 시에도 chromium 접근 가능. `/app/data` 는 `a+rwX`, `/app` 나머지는 `a+rX` 로 단독 컨테이너 실행 시에도 런타임 쓰기 허용.
  - **subprocess 규약**: `subprocess.Popen(..., start_new_session=True)` 로 새 프로세스 그룹 (`os.setsid`) 생성 → 중단 시 `os.killpg(pgid, SIGTERM)` 으로 그룹 전체 전파 → compose v2 가 관리 컨테이너 PID 1 로 릴레이. detach 금지. stdout/stderr 는 **파일 redirect**(`data/logs/scrape_runs/{run_id}.log`) — PIPE 는 버퍼 가득 차면 자식이 블로킹됨.
  - **entrypoint 설정 주입**: `docker/entrypoint.sh` 가 마운트된 `sources.yaml` 을 per-run 임시 디렉터리(`mktemp`)로 복사하고 `SOURCES_CONFIG_PATH` 환경변수를 주입한 뒤 `python -m app.cli` 를 exec. **sources.yaml 우선순위**: (1) compose 바인드 마운트 → (2) 이미지 내 `sources.yaml.template` 폴백. 실행 파라미터는 `sources.yaml`의 `scrape:` 섹션으로 제어하되, 웹/스케줄러가 "이번 run 만 특정 소스" 로 제한할 때는 `SCRAPE_ACTIVE_SOURCES`(쉼표 연결 문자열) 환경변수로 전달한다 — 파일 수정을 피해 per-run 복사본 격리를 유지한다.   - **동시성 lock**: ScrapeRun 테이블의 `running` row 존재 시 신규 생성 거부. create_scrape_run 이 단일 트랜잭션 안에서 SELECT→INSERT 해 race 방지. 웹 startup 에서 `cleanup_stale_running_runs()` 가 pid 없는/프로세스 사라진 running row 를 failed 로 정리해 lock 자동 해제. 이후 `start_scheduler()` 순서 고정.
  - **중단(SIGTERM) 시 atomic 보장**: `app/cli.py` 에 SIGTERM handler 등록 → 모듈 레벨 `_cancel_flag` 만 세움. 공고 루프 / 소스 루프의 **시작 지점에서만** flag 확인해 break. 현재 처리 중 공고(첨부 다운로드 포함)는 끝까지 마무리 — Phase 1a 의 per-announcement atomic 트랜잭션(UPSERT + 변경 감지 + 사용자 라벨링 리셋)을 깨지 않기 위함. ScrapeRun.status 도메인: `running/completed/cancelled/failed/partial`.
- **초기 설정**: `sources.yaml`은 `.gitignore` 대상(브랜치 전환 시 보존). 최초 기동 전 `sh scripts/bootstrap_sources.sh`(또는 `cp -n sources.yaml.template sources.yaml`)로 생성 후 편집. `sources.yaml.template`이 이미지에도 포함되어 있어 마운트 누락 시 기본값으로 기동 가능(경고 로그 출력).
- **Dockerfile 구성**: `alembic.ini`·`alembic/` 디렉터리·`sources.yaml.template`·`scripts/` 디렉터리를 이미지에 COPY. `entrypoint.sh`가 `alembic upgrade head`를 실행하므로 migration 산출물이 이미지 내에 반드시 포함되어야 한다. `scripts/`는 `create_admin.py` 등 운영자 CLI를 컨테이너 내부에서 실행하기 위해 포함 — compose 바인드 마운트(`./scripts:/app/scripts`)와 병행 적용.
- **외부 의존성**: httpx(목록·상세 수집), BeautifulSoup4(상세 HTML 파싱), pyyaml(sources.yaml 로드), SQLAlchemy 2.0, **Alembic 1.x**(스키마 마이그레이션), FastAPI, uvicorn[standard], Jinja2, pydantic 2 + pydantic-settings, python-dotenv, loguru, **playwright**(첨부 다운로드 — headless Chromium), **passlib[bcrypt] + bcrypt**(Phase 1b 비밀번호 해시), **python-multipart**(FastAPI Form/File 파싱), **APScheduler 3.x**(Phase 2/00025 — 웹 프로세스 내부 BackgroundScheduler; 4.x 는 async-first 로 API 가 breaking change 여서 3.x 로 고정). Dockerfile 에 `playwright install chromium --with-deps` + `docker.io` + `docker-cli`(apt — 두 패키지 분리됨, 모두 필요) + compose v2 플러그인(직접 curl 배치) 포함.
- **첨부 다운로드 설계**: `docs/attachment_download_plan.md` 에 상세 설계 기록. 첨부 링크는 `javascript:f_bsnsAncm_downloadAtchFile(...)` 형태 — Playwright가 해당 링크를 클릭하면 브라우저 `download` 이벤트로 파일 수신. httpx fallback(직접 POST)은 JS 우회 불가 시 사용 못할 수 있음.

## 컨벤션

- **언어/문서**: 모든 모듈 docstring과 주석은 **한국어**. 외부 공개용이 아니므로 격식보다 명확성 우선.
- **타입힌트**: 모든 공개 함수에 타입 어노테이션. `from __future__ import annotations` 를 파일 상단에 둔다.
- **설정 접근**: 직접 `os.environ` 읽지 말고 `app.config.get_settings()` 싱글턴을 통해 접근.
- **시각 처리**: DB의 모든 시간 컬럼은 timezone-aware UTC. 소스가 모호하면 timezone 추정 없이 UTC로 보존하고 원문 텍스트는 `raw_metadata` JSON에 함께 저장한다. 단, **SQLite 백엔드는 `DateTime(timezone=True)` 의 tz 정보를 실제로 저장하지 못해** SELECT 결과가 naive `datetime` 으로 돌아온다. tz-aware 시각(`datetime.now(tz=UTC)` 등)과 직접 비교하면 `TypeError`가 발생하므로, 비교 직전에 양쪽 값을 모두 UTC tz-aware 로 정규화하는 헬퍼(예: `app/auth/service._as_utc`)를 사용한다. 저장 컨벤션이 UTC라는 전제에 의존한다.
- **UPSERT**: 공고는 `(source_type, source_announcement_id)` 복합 키 기준, 첨부는 `(announcement_id, original_filename)` + `sha256` 비교. 재실행 시 중복 생성 금지. 변경된 경우에만 신규 row 생성(이력 보존), 상태 전이만인 경우는 in-place UPDATE.
- **예외 격리**: 스크래퍼 파이프라인은 **공고 1건 단위**로 try/except. 한 공고의 실패가 전체 실행을 중단시키지 않는다.
- **웹 보안 경계**: FastAPI 는 외부 노출 금지가 여전히 전제 — 인증이 있지만 로컬 전용 UI 용 가벼운 계층이다(Secure=False 쿠키, CSRF 토큰 없음). 첨부 다운로드는 반드시 `download_dir` 하위로 경로 트래버설 방어. POST 인증 라우트는 `ensure_same_origin` 의존성으로 Origin/Referer 체크만 적용.
- **로깅**: loguru + stdlib 브리지(`InterceptHandler`). 레벨은 `LOG_LEVEL` 환경변수(`DEBUG/INFO/WARNING/ERROR/CRITICAL`). 웹 프로세스는 `create_app()` 최상단에서 `configure_logging()` 1회 호출 필수. uvicorn/starlette 계열 로거는 자체 StreamHandler를 제거하고 propagate=True로 전환해 중복 출력 방지. sqlalchemy·alembic은 DEBUG 시에도 최소 INFO 고정. 모든 요청/응답은 `LoggingMiddleware`가 기록, 미처리 예외는 전역 핸들러(`observability.py`)가 stack trace 포함 강제 출력.
- **패키지 경로 컨벤션**: uvicorn 경로는 `app.web.main:app`, 스크래퍼 CLI는 `python -m app.cli`(서브커맨드 없음, CLI 인자 없음). 재구조화 시 `docker/entrypoint.sh`와 compose의 command도 함께 조정 필요.
- **Jinja2 `<script>` 블록 내 태그 리터럴 금지**: Jinja2 파서는 HTML `<script>` 블록 안의 JavaScript 주석까지 포함해 `{% %}` 태그를 해석한다. `script` 블록 주석에서 Jinja 태그 문자열을 언급해야 할 때는 `{% else %}` 같은 리터럴 대신 "else 분기" 등 플레인 텍스트로 대체한다. 위반 시 `TemplateSyntaxError`로 해당 페이지 전체 500 에러 발생(00031 사례).
- **로컬 Python 직접 실행 금지**: 호스트에서 `python -m app.cli` 실행은 지원하지 않는다. Docker Compose 경유만 지원. README에서도 관련 예시 모두 제거됨.
- **lint/format**: ruff (line-length 110, target py311, E501 무시). mypy는 옵션 dev 의존성으로만 설치.
- **N+1 제거 패턴 (목록 페이지)**: 로그인 사용자 종속 데이터(읽음·관련성)는 목록 announcement_id/canonical_id 전체를 `IN` 절로 한 번에 SELECT해 dict로 반환하는 `get_X_map(ids)` 헬퍼로 일괄 조회 후 템플릿에 주입. 목록 당 추가 쿼리는 2~3개로 고정. Phase 1b `get_read_announcement_id_set` → Phase 3a `get_relevance_by_canonical_id_map` / `get_relevance_history_by_canonical_id_map` 동일 패턴.
- **Jinja2 파셜/매크로 재사용 패턴**: 위젯 단위 UI(`_relevance_badge_macro.html`, `_relevance_modal.html`)를 `templates/` 루트에 `_` 접두사 파셜로 두고 `{% from %}` / `{% include %}` 로 목록·상세 양쪽에서 재사용. 동일 위젯이 두 페이지에 필요할 때 파셜 분리가 기준.
- **테스트**: CI 자동 실행은 비활성화 상태(`config_override.testing.*.enabled=false`). 단, `tests/db/` 아래 유닛 테스트가 실제 존재함 — 변경 감지 케이스별 시나리오(`test_change_detection.py`), 리셋·트랜잭션 atomic rollback(`test_atomic_rollback.py`), conftest에 인메모리 SQLite + 가짜 User fixture. Phase 3a(00035)에서 `tests/bulk/`(`test_bulk_repository.py`, `test_bulk_routes.py`) + `tests/relevance/`(`test_relevance_repository.py`, `test_relevance_routes.py`) 추가. 수동으로는 `pytest tests/` 로 실행 가능.
- **커밋 메시지**: `[{task_id}][tg:{requester}] {subtask_id}: {요약}` 형식으로 통일.
- **데이터/비밀**: `.env` 와 `sources.yaml` 은 커밋 금지(`.gitignore` 대상). `.env.example`·`sources.yaml.template` 만 관리. `sources.yaml` 초기 생성은 `sh scripts/bootstrap_sources.sh`.
- **Migration/ORM 이식성**: ORM 컬럼 타입은 JSON(범용·JSONB 금지), DateTime(timezone=True), String(N) dialect 중립 타입만 사용. ALTER TABLE은 `batch_alter_table`로 감싸기(SQLite 호환 필수). 각 migration은 `upgrade()` + `downgrade()` 양방향 구현 필수. 대소문자 무시 검색은 `func.lower() + LIKE`. 상세 규칙은 `docs/db_portability.md` 참조.
- **Migration 검증 절차**: 신규 migration 추가 시 기존 DB stamp 무변경 + 빈 SQLite bootstrap + Postgres syntax 3단계 검증. 절차는 `docs/alembic_verification.md` 참조.

## 주요 결정

- **목록·상세 수집은 httpx(HTTP-only), 첨부 수집만 Playwright**: IRIS 목록·상세 페이지는 SSR이라 httpx만으로 충분(탐사로 확인됨). 첨부 링크는 `javascript:` onclick으로만 노출되어 HTTP 직접 접근 불가 → Playwright headless로만 처리. 이 구분이 수집 단계 안정성을 높이고 의존성 무게를 줄인다.
- **Alembic 도입 완료 (00017)**: `alembic/versions/` 아래 migration 파일로 스키마를 관리. `init_db()` 는 Alembic Python API(`alembic.command.upgrade/stamp`)를 사용하여 stamp vs upgrade 분기를 자동 선택. 기존 DB는 `stamp head` 로 데이터 무변경, 신규 DB는 `upgrade head` 로 전체 스키마 생성. `env.py` 는 URL을 `alembic_config`에서 우선 읽어 호출자가 주입한 엔진을 존중한다.
- **SQLite 기본 + `DB_URL` 로 교체 가능**: 배포/공유 부담이 없는 단일 파일 DB가 개인용 목적에 맞음. Postgres 전환은 접속 문자열만 바꾸면 되도록 ORM 쪽은 JSON 범용 타입 사용.
- **첨부 바이너리는 DB 밖 FS에 저장**: hwp/zip이 수십 MB가 될 수 있어 BLOB 컬럼은 부적절. DB엔 경로/해시만. 중복/변경은 `sha256` 으로 판정.
- **페이지/공고 수집 상한을 sources.yaml·코드 기본값 2단계 우선순위로 관리**: 코드 기본값은 `max_pages=10, max_announcements=200`. 우선순위: `sources.yaml scrape:` 섹션 전역값 > 소스별 설정 > 코드 default. CLI 인자 단계는 00016에서 제거됨. 기본값을 보수적으로 유지해 차단 회피를 기본 정책으로 삼는 기조는 유지.
- **FastAPI 로컬 전용 + 세션 쿠키 인증(Phase 1b, 00021 에서 도입)**: "개인용 → 팀 공용" 로드맵에 따라 auth 를 도입했으나 전제는 여전히 로컬 네트워크 한정이다. **비로그인 열람은 유지** — 목록/상세 URL 은 기존 그대로이고, `current_user_optional` 로 None 이면 읽음 로직을 skip. 자유 회원가입(관리자 승인·이메일 검증 X), 비밀번호 bcrypt(passlib, rounds=12), 서버 사이드 세션(`UserSession` row) + HttpOnly/SameSite=Lax 쿠키(Secure=False, 로컬 HTTP 전제), 세션 ID `secrets.token_urlsafe(32)` 평문 DB 저장. JWT 는 의도적으로 선택하지 않았다(로컬 혁신성 대비 운영 복잡도 증가). 정책 상수와 근거는 `app/auth/constants.py` 에 주석과 함께 집중.
- **상태값을 한글 원문 그대로 Enum 값으로 보존**: IRIS 노출 텍스트와 1:1 매칭되어 필터/표시 변환 비용을 줄이기 위함.
- **증분 수집 전략 — DB 초기화 금지**: 스크래핑 재실행 시 기존 DB를 유지하고 변경된 공고만 갱신. `init_db()`는 Alembic stamp/upgrade로 스키마만 보장하고 기존 데이터를 절대 건드리지 않음. 이력 보존 및 불필요한 세부 수집 방지가 목적.
- **`received_at`을 변경 감지 비교 필드에서 제외**: 접수예정 상태에서 수집 시 공란이었다가 이후 보완되는 경우가 많아 잡음 트리거를 피하기 위해 의도적으로 제외. 핵심 비교 필드는 `title, status, deadline_at, agency` 4개로 고정.
- **2차 변경 감지 도입 (00019)**: 1차 감지(목록 UPSERT 시)는 sha256을 알 수 없어 첨부 변경 탐지 불가. 첨부 다운로드 완료 후 2차 감지를 추가해 첨부 개수·sha256 변동을 감지. `_upsert_announcement`를 비교 필드 파라미터화해 두 단계가 동일 함수를 재사용.
- **사용자 라벨링 데이터 보호가 변경 감지 리셋 정책의 핵심 근거**: 공고 내용이 실질 변경되면 `AnnouncementUserState(is_read)` 및 `RelevanceJudgment` 이력을 재설정 — 사용자 판단이 변경 전 내용 기준으로 오염되지 않도록. FavoriteEntry는 내용과 무관하게 유지.
- **FavoriteFolder depth 2 제약은 ORM validator**: SQLite ALTER TABLE 제약 추가 어려움과 단순 계층 구조 요구사항을 감안, DB CHECK 대신 ORM `@validates` 레벨에서 depth 검증.
- **Docker `scraper` 서비스를 `profiles: [scrape]` 로 격리**: 스크래퍼는 1회성 배치라 기본 `compose up` 에서 기동되면 안 됨. 명시적 `--profile scrape run --rm` 으로만 실행.
- **날짜 기반 수집 중단 조건 제거**: 정부 공고 사이트가 마감일을 `2200-01-01` 등 허위 값으로 기재하는 사례가 많아 날짜로 중단 여부를 판단하면 오작동 가능성이 큼. `list_scraper` 에서 날짜 기반 break 조건을 전부 제거하고 `max_pages` / `max_announcements` 상한만으로 수집을 제어한다.
- **외부 CDN 의존 없는 정적 리소스**: 오프라인/격리 환경에서도 UI가 그대로 동작하도록 CSS를 `app/web/static/` 패키지 내부에 둔다.
- **Canonical 스키마 — (B) 별도 테이블 선택**: `announcements`에 컬럼을 추가하는 (A)보다 `canonical_projects` 별도 테이블 + FK 참조(N:1) 구조(B)를 선택. 이유: cross-source 그룹 단위 검색·UI 표시 시 JOIN 없이 그룹 메타를 바로 조회할 수 있고, 나중에 그룹 수준 필드를 확장하기 쉽다.
- **IRIS 공식 식별자는 `ancmNo`**: 목록 API 전수 필드에서 비어 있는 레코드 0건. 1:1 고유 키가 아니라 "공고 묶음" 단위(하나의 공고번호에 여러 세부과제 ancmId가 대응). canonical_key 생성 전 NFKC 정규화 + 공백 제거 필수. 탐사 결과는 `docs/canonical_identity_design.md §2` 참조.
- **`compute_canonical_key` 공식 키 우선 + fuzzy fallback**: official 키(ancmNo 정규화)를 우선 사용하고, 없으면 (제목 불용어 제거 + 주관기관 정규화 + 마감연도) 조합. NTIS cross-source 실매칭에서도 동일 로직 재사용 확인(fuzzy 매칭 → official 승급 경로 검증됨). fuzzy 임계값·튜닝 근거는 `docs/canonical_identity_design.md` 참조.
- **NTIS는 httpx 단독 수집 (Playwright 불필요)**: 목록·상세·첨부 모두 httpx POST SSR로 처리. IRIS와 달리 AJAX API 없음. 첨부도 `wfUid + roTextUid` 파라미터로 httpx POST 직접 다운로드 가능(JS onclick 우회 불필요). `AttachmentDownloader`는 소스별 링크 추출 분기를 adapter 계층에서 처리하고 downloader 자체는 소스를 모름.
- **sources.yaml에 credentials 슬롯(optional)**: 로그인이 필요한 소스를 위한 `credentials.username`/`credentials.password` 필드를 `SourceConfig`에 예약. NTIS는 게스트 수집이 가능하므로 현재 미사용. 실제 값은 `.env`로만 주입하고 yaml에는 키 이름만 남긴다.
- **CSRF 토큰 대신 same-origin 체크(Phase 1b)**: 로컬 팀 전용 전제라서 CSRF 토큰 기반 방어는 과잉. POST 인증 라우트는 `ensure_same_origin` 의존성으로 Origin/Referer 헤더만 netloc 비교한다. 헤더가 없는 비-브라우저 호출(curl 등)은 통과시켜 스크립트 운영 마찰을 줄인다.
- **읽음/안읽음 UI 는 로그인 시만 활성(Phase 1b)**: `AnnouncementUserState.is_read` 조회를 위해 현재 페이지 `announcement_id` 집합을 한 번에 SELECT(`get_read_announcement_id_set`) 해 N+1 을 제거. 상세 진입 시 `mark_announcement_read` 로 UPSERT. 비로그인은 전체 skip — 기본 경로에 오버헤드 0. IRIS/NTIS 는 announcement 단위로 독립 관리(canonical 묶음에 종속시키지 않음) — 같은 공고의 소스별 버전을 각각 읽음 처리 가능하도록.
- **첫 관리자 계정은 CLI 로만 생성**: 자유 회원가입은 항상 `is_admin=False`. 관리자 UI 는 Phase 2+ 이므로, 그때까지는 `scripts/create_admin.py` (getpass 2회 + argparse username/--email) 를 통해서만 관리자를 생성한다. 컨테이너 내부 TTY 실행을 전제로 설계.
- **sources.yaml을 gitignore 전환 + template 도입 (00022)**: sources.yaml이 git에 커밋되어 있으면 브랜치 전환 시 로컬 수정본이 롤백되는 문제가 있다. sources.yaml을 `.gitignore`에 추가하고, `sources.yaml.template`(플레이스홀더 포함)만 버전 관리. 이미지에도 template을 COPY해 두어 마운트 누락 시 폴백으로 사용.
- **Dockerfile에 alembic 산출물 포함 필수 (00022)**: `alembic.ini`와 `alembic/` 디렉터리가 이미지에 없으면 `entrypoint.sh`의 `alembic upgrade head`가 "No 'script_location' key found" 오류로 실패한다. 이미지 빌드 시 반드시 COPY해야 하며, migration 버전 추가 시 이미지 재빌드 필요.
- **웹 수집 제어는 docker-in-docker 대신 호스트 docker.sock 원격 조작 (00025)**: `dind` 방식은 이미지·권한·볼륨 재설계 부담이 커서, app 컨테이너에 호스트 `/var/run/docker.sock` 을 마운트하고 docker CLI + compose v2 플러그인을 설치해 `docker compose` 를 호출하는 방식을 선택. 호스트 dockerd 가 실제 컨테이너를 관리하므로 scraper 의 `./data`·`./app`·`./sources.yaml` 바인드가 기존 CLI 경로와 동일하게 적용된다. 대가: docker.sock 마운트는 사실상 호스트 root 권한 부여와 동등 — FastAPI 외부 노출 금지 전제가 이 구성에서 더 강하게 요구된다. 설계 근거는 `docs/scrape_control_design.md §5`.
- **compose v2 필수 (docker-compose v1 금지, 00025)**: v2 가 SIGTERM 을 관리 컨테이너 PID 1 로 공식 전파한다. v1 `docker-compose` 는 전파 동작이 검증되지 않아 "중단 → 공고 마무리 후 cancelled" 흐름이 깨질 수 있다. Dockerfile 이 compose v2 바이너리를 `/usr/libexec/docker/cli-plugins` 에 직접 배치해 `docker compose` 서브커맨드 형태로만 호출한다.
- **subprocess 는 새 프로세스 그룹 + 파일 redirect (00025)**: 웹/스케줄러 → scraper subprocess 기동 시 `start_new_session=True` (`os.setsid`) 로 새 프로세스 그룹 생성. 중단은 `os.killpg(pgid, SIGTERM)` 로 그룹 전체 전파 — compose v2 가 컨테이너 PID 1 로 릴레이. detach/데몬화 금지(부모-자식 신호 체인이 끊김). stdout/stderr 는 PIPE 가 아닌 `data/logs/scrape_runs/{run_id}.log` **파일 redirect** — PIPE 버퍼가 가득 차면 자식이 write 에서 블로킹된다. Windows 호환은 포기(`os.setsid` POSIX 전용).
- **ScrapeRun 을 세 경로 공통 lock 으로 사용 (00025)**: 웹(trigger='manual')/스케줄러(trigger='scheduled')/CLI(trigger='cli') 모두 `create_scrape_run` 이 단일 트랜잭션 안에서 "running row 존재 검사 → INSERT" 를 수행. running row 가 곧 "실행 중 flag" — 별도 파일 락·메모리 뮤텍스를 도입하지 않는다. 웹 startup 에서 `cleanup_stale_running_runs()` 가 pid 없는/프로세스 사라진 running row 를 failed 로 정리해 lock 자동 해제. APScheduler `start_scheduler()` 는 cleanup 이후에 호출해 race 를 피한다.
- **`SCRAPE_RUN_ID` 환경변수로 웹→CLI subprocess ScrapeRun 소유권 이전 (00034)**: 웹/스케줄러의 `start_scrape_run` 은 `create_scrape_run` 으로 running row 를 먼저 INSERT 한 뒤 subprocess 를 기동한다. 기존 CLI 가 그대로 "running row 존재 검사" 를 수행하면 방금 생성된 자기 row 를 '이미 진행 중인 다른 수집' 으로 오판해 exit(2) 하는 자기참조 버그가 있었다. 해결: `SCRAPE_RUN_ID` 환경변수로 id 를 주입하면 CLI 는 새 row 를 만들지 않고 해당 row 를 이어받아 finalize 까지 수행한다. 미주입이거나 파싱 실패 시 기존 CLI 직접 실행 경로(trigger='cli' 로 자체 create_scrape_run) 로 동작해 하위 호환이 유지된다.
- **SIGTERM 처리는 atomic 경계 보존 (00025)**: CLI 가 SIGTERM 을 받으면 handler 는 모듈 레벨 `_cancel_flag` 만 세우고, 실제 break 는 **공고 루프/소스 루프의 시작 지점에서만** 수행. 현재 공고(첨부 다운로드 포함)는 끝까지 마무리 — Phase 1a 에서 확립한 "UPSERT + 변경 감지 + 사용자 라벨링 리셋은 같은 트랜잭션" 원칙을 중단 경로에서도 깨지 않기 위함. 종료 status 는 `cancelled`.
- **APScheduler 3.x 를 웹 프로세스 내부 BackgroundScheduler 로 (00025-6)**: 별도 worker 프로세스를 띄우지 않고 FastAPI 와 같은 프로세스 안에서 `BackgroundScheduler` 를 가동 — 운영 단순성이 팀 공용 로컬 배포 목적에 부합. JobStore 는 `SQLAlchemyJobStore` 단일(`scheduler_jobs` 테이블, APScheduler 가 자체 생성해 Alembic 관리 밖). 재시작 시 잡 자동 복원. `misfire_grace_time=300`·`coalesce=True`·`max_instances=1` 로 놓친 잡 합치기 및 중복 기동 방지. 4.x 는 API breaking change 라 `apscheduler>=3.10,<4.0` 으로 고정.
- **SCRAPE_ACTIVE_SOURCES 환경변수로 per-run 소스 override (00025)**: 웹 UI "지금 시작(source 선택)" 은 `sources.yaml` 파일을 in-place 수정하면 entrypoint 의 per-run 복사본 격리가 깨져 동시 실행 시 설정 경합이 난다. 파일 대신 `SCRAPE_ACTIVE_SOURCES` 환경변수(쉼표 연결 문자열, 예 `IRIS,NTIS`)로 주입 — `app/cli.py` 가 로드 후 우선 적용, 미설정/빈 값이면 `sources.yaml` 의 `active_sources` 를 그대로 사용.
- **컨테이너를 호스트 UID/GID 로 실행 (00028)**: `docker-compose.yml` 두 서비스에 `user: "${HOST_UID}:${HOST_GID}"` 적용. `.env` 에 명시 설정 필수(fallback 없음). 목적: `./data/` 하위 모든 런타임 생성 파일이 호스트 유저 소유가 되어 `sudo` 없이 관리 가능. 기존 root-owned 파일은 `sudo chown -R "$(id -u):$(id -g)" ./data/` 로 일회성 복구.
- **sources.yaml 웹 편집기는 원자적 쓰기 + 백업 필수 (00025-5)**: 편집 흐름은 "YAML syntax → `SourcesConfig.model_validate` 통과 시에만 저장". 저장 전 `data/backups/sources/YYYYMMDD_HHMMSS.yaml` 로 타임스탬프 백업. 쓰기는 `NamedTemporaryFile` + `os.replace` 원자적; 두 경로에서 `_overwrite_with_fsync` fallback: (a) EXDEV(bind mount 파일 단위, tmp 와 target 다른 device), (b) EACCES/EPERM(`/run/config/` 부모 dir 가 비쓰기 tmpfs 라 NamedTemporaryFile 생성 불가 — compose 표준 구성에서 발생하는 전형 케이스). 편집 대상은 bind mount 원본이라 현재 실행 중인 subprocess 의 per-run 복사본에는 영향 없음 — **다음** 수집부터 반영.
- **stdlib→loguru 브리지가 필수인 이유 (00030)**: uvicorn은 FastAPI 라우트에서 발생한 미처리 예외를 stdlib `logging`의 `uvicorn.error` 채널로 기록한다. loguru는 stdlib와 별도 시스템이라 브리지(`InterceptHandler`) 없이는 이 stack trace가 docker logs에 전혀 나타나지 않는다. LOG_LEVEL=DEBUG 설정에도 불구하고 로그가 비어 보이던 근본 원인이 이것이었다. 해결: `InterceptHandler`를 root logger에 부착 + uvicorn 로거 핸들러 명시적 교체.
- **APScheduler 백그라운드 컨텍스트에서 docker 바이너리를 절대경로로 탐색 (00029)**: `job_runner.py` 의 `scheduled_scrape` 가 호출하는 `build_compose_command` 는 `argv[0]` 을 문자열 `"docker"` 로 두면 APScheduler 실행 컨텍스트의 제한된 PATH 에서 `FileNotFoundError` 가 발생한다. `_resolve_docker_binary()` 를 도입해 `shutil.which` → 관행 경로(`/usr/bin/docker`, `/usr/local/bin/docker`) → 환경변수 `DOCKER_BINARY` 순으로 탐색하고 절대경로를 반환. 탐색 실패 시 `ComposeEnvironmentError` 로 ScrapeRun 을 failed 처리해 UI 에 진단 메시지를 노출한다.
- **`as_utc` 헬퍼를 auth 전용 → 공용으로 승격 (00025)**: `_as_utc` 로 `app/auth/service.py` 에만 있던 helper 를 `app/db/models.py` 로 이동해 `scrape_runs` 비교 로직에서 재사용. SQLite 의 `DateTime(timezone=True)` tz 손실은 프로젝트 공통 이슈라 컬럼 비교가 있는 모든 경로에서 동일 패턴이 필요하다.
- **관련성 단위는 canonical, 읽음 단위는 announcement — 혼용 금지 (00035)**: `RelevanceJudgment`의 unique 키는 `(canonical_project_id, user_id)` — 여러 소스의 announcement가 같은 canonical이면 판정을 공유한다. 반면 `AnnouncementUserState.is_read`는 `announcement_id` 단위 유지 — IRIS/NTIS 각 공고를 별도 읽음 처리 가능하도록. 이 두 단위는 의도적으로 다르며, 목록 row는 announcement이지만 관련성 배지는 canonical_id로 조회·표시한다.
- **관련성 판정 변경은 원자적 이관 (user_changed) — 덮어쓰기 금지 (00035)**: 기존 판정이 있을 때 새 판정을 저장하면 기존을 `RelevanceJudgmentHistory`로 이관(archive_reason="user_changed")한 뒤 신규 판정을 INSERT — 동일 트랜잭션. 직접 UPDATE로 덮으면 이력이 소실되므로 설계상 금지. content_changed 이관(공고 내용 변경 시)과 동일 패턴.
- **bulk 요청 두 가지 모드: ids vs filter (00035)**: 페이지 내 선택은 `ids` 모드(명시 목록), "현재 필터 결과 전체 M건 선택"은 `filter` 모드(서버가 필터 파라미터로 announcement id를 추출). `MAX_BULK_MARK` 상수로 한 번에 처리 가능한 상한을 제한. 클라이언트가 필터 재구성을 서버에 위임해 페이지를 넘는 대량 선택을 지원.
- **배지 클릭 vs row 클릭 충돌은 stopPropagation으로 해결 (00035)**: 목록 row 전체가 상세 링크(`<a>`)로 감싸져 있을 때 내부 배지 클릭이 상세 페이지로 이동하지 않도록 `event.stopPropagation()` + `preventDefault()` 패턴 적용. 이후 클릭 이벤트가 있는 인라인 위젯 추가 시 동일 패턴 사용.
