# PROJECT NOTES

## 프로젝트 개요

IRIS(한국연구재단 범부처통합연구지원시스템)·NTIS 등 정부과제 사업공고를 자동 수집·저장하고 로컬 웹에서 목록/상세/첨부파일을 열람하게 해주는 **개인용 로컬 에이전트**. 첨부파일(pdf/hwp/hwpx/zip) 에 상세 정보가 담겨 있으므로 첨부 수집이 핵심이다. IRIS·NTIS 모두 완전 구현 상태이며, 두 소스 모두 **접수예정·접수중·마감 3개 상태 전체**를 순차 수집한다.

운영 가이드는 [README.USER.md](README.USER.md), 프로젝트 개요·설치는 [README.md](README.md) 참조. 본 문서는 새 개발자가 "어떤 모듈이 있고 무엇을 하며 어떤 컨벤션·결정이 살아 있는지" 파악하기 위한 아키텍처 / 컨벤션 / 결정 기록이다.

## 아키텍처

### 패키지 구조 (`app/`)

#### 코어 / 인프라

- `app/cli.py` — 스크래퍼 오케스트레이터 엔트리포인트(`python -m app.cli`). CLI argparse 없음. `sources.yaml` 의 `scrape:` 섹션에서 실행 파라미터를 읽고 enabled/active 소스를 순서대로 실행. 우선순위: `scrape:` > 소스별 설정 > 코드 default. SIGTERM handler 등록 — 모듈 레벨 `_cancel_flag` 만 세우고 break 는 공고 루프 / 소스 루프 시작 지점에서만 수행(현재 처리 중 공고는 끝까지 마무리해 atomic 보장).
- `app/config.py` — pydantic-settings 기반 `Settings` 싱글턴(`get_settings()`). `.env` 로더, 런타임 디렉터리 생성(`ensure_runtime_paths`). 소스별 URL 은 `sources.yaml` 에서만 관리.
- `app/sources/` — `constants.py`(소스 ID 상수), `config_schema.py`(`ScrapeRunConfig`/`SourceConfig`/`SourcesConfig` Pydantic 모델), `load_sources_config()`. `SOURCES_CONFIG_PATH` 환경변수 경로를 우선 사용(entrypoint 의 per-run 임시 복사본 격리용). `app/sources/yaml_editor.py` — 관리자 [sources.yaml] 탭의 편집 서비스. YAML 파싱 → `SourcesConfig.model_validate` → 백업(`data/backups/sources/YYYYMMDD_HHMMSS.yaml`) → `NamedTemporaryFile` + `os.replace` 원자적 쓰기. EXDEV / EACCES 케이스에는 `_overwrite_with_fsync` fallback. 편집 대상은 bind mount 원본이므로 실행 중인 subprocess 의 per-run 복사본에는 영향 없고 다음 수집부터 반영된다.
- `app/timezone.py` — KST 단일 운영 헬퍼. `KST = ZoneInfo("Asia/Seoul")`. `to_kst(dt)` 는 UTC tz-aware / naive(UTC 가정) / KST tz-aware 어떤 입력도 KST tz-aware 로 정규화하고 None 통과. `now_kst()` / `now_utc()` (저장은 utc, 표시는 kst), `format_kst(dt, fmt)`, `kst_date_boundaries(date)` (KST 일자의 [00:00, 24:00) 구간을 UTC tz-aware 로 반환). `_as_utc`(SQLite SELECT 보정) 와 `to_kst`(KST 변환) 는 별개 레이어이며 결합 사용 (`_as_utc` → `to_kst`).
- `app/logging_setup.py` — loguru 루트 로거 초기화 + stdlib→loguru 브리지(`InterceptHandler`). uvicorn / starlette / FastAPI / SQLAlchemy / alembic 서드파티 로그도 동일 sink 로 수렴. `configure_logging()` 은 idempotent. `_NOISY_LOGGER_NAMES`(sqlalchemy.engine 등) 는 DEBUG 모드에서도 최소 INFO 로 고정. sink format 은 KST 타임스탬프 명시.
- `app/web/observability.py` — HTTP 요청·응답 미들웨어(`LoggingMiddleware`, `method path → status_code (Xms)` 기록) + 전역 예외 핸들러(미처리 예외를 `logger.opt(exception=True).error()` 로 강제 stack trace 출력 후 500 응답).
- `app/web/access_log.py` / `app/web/access_log_stats.py` — IP 접근 이력 ASGI 미들웨어와 집계 유틸. 요청에서 `ip_address·accessed_at·path·method·user_agent·user_id·status_code` 7개를 추출해 KST 날짜별 `access_history_YYMMDD.log`(JSON-lines) 에 append. `/static/`, `/admin/scrape/status`, `/favicon.ico`, `/health[z]` 는 skip. 저장 경로는 `ACCESS_LOG_DIR` 환경변수. 집계 유틸은 일별·IP별·세션 집계(`SESSION_GAP_MINUTES` 기본 30분)와 `filter_records_by_ip(records, ip_list, mode)`(include / exclude 필터) 를 제공.

#### 데이터 / 도메인

- `app/db/` — SQLAlchemy 2.0 ORM. `models.py` / `session.py` / `repository.py` / `init_db.py` / `migration.py` / `snapshot.py`. 공고 UPSERT 키는 `(source_type, source_announcement_id)` 복합 키. 스키마 마이그레이션은 **Alembic** (`alembic/versions/`) 전담 — `init_db()` 가 stamp vs upgrade 분기를 자동 선택. `snapshot.py` 의 `merge_snapshot_payload(existing, new)` 가 5종 카테고리 머지 로직과 `ScrapeSnapshot` UPSERT(KST 날짜당 1 row) 를 담당. 테이블 군은 크게 (a) 도메인(`announcements`, `attachments`, `canonical_projects`), (b) delta 파이프라인(`delta_announcements`, `delta_attachments`, `scrape_snapshots`, `scrape_runs`), (c) 사용자(`users`, `user_sessions`, `announcement_user_states`, `relevance_judgments` + `*_history`, `announcement_progress` + `announcement_progress_history`, `favorite_folders` + `favorite_entries`, `email_subscriptions`, `audit_logs`), (d) 조직(`organizations`, `user_organizations`), (e) 시스템(`system_settings` key-value 설정 영속, `backup_history` 백업 실행 이력, `email_send_runs` 메일 발송 이력, `email_forward_logs` 공고 포워딩 이력, `email_daily_report_runs` 단체 Daily Report 발송 실행 이력). `system_settings` 의 `email.*` 네임스페이스(메일 발송 설정 7개 키 + Daily Report 설정 4개 키)는 메일 관련 설정 영속용. 스키마 상세는 `docs/schema_phase1a.md`.
- `app/canonical.py` — `compute_canonical_key()`. 공식 키(ancmNo + 공고명 합성, `official:{normalized_ancmNo}::{normalized_title}`) 우선, 없으면 (제목 + 기관 + 마감연도) fuzzy fallback. 결과는 frozen dataclass `CanonicalKeyResult(canonical_key, canonical_scheme)`. 정규화 규칙은 `docs/canonical_identity_design.md §4-1`.
- `app/auth/` — 세션 쿠키 인증. `constants.py`(쿠키·세션·비밀번호 정책), `service.py`(passlib bcrypt 해시 + `create_user`/`authenticate`/`create_session`/`get_active_session`/`delete_session` + `change_password`/`change_email`/`set_email_subscribed`/`delete_user` cascade), `dependencies.py`(`current_user_optional`/`current_user_required`/`admin_user_required` + `ensure_same_origin` CSRF-lite 체크), `routes.py`(`/register`, `/login`, `/auth/register|login|logout|me`). 비로그인 열람 유지 — `current_user_optional` 이 None 이면 사용자 종속 로직만 skip.
- `app/organizations/` — 조직 트리 + M:N 매핑 서비스. `service.py` (`get_organization_tree`/`create_organization`/`delete_organization`/`set_user_organizations`/`rename_organization`(동일 부모 동명 차단)/`move_organization`(자기 자신·후손으로 이동 차단, `_descendant_ids` BFS)), `io.py` (`export_organization_tree_json` / `import_organization_tree_json` — DB 전체 교체 + user_organizations 이름 경로 기반 재매핑). 트리 조회는 재귀 CTE 대신 전체 SELECT → 메모리 빌드(parent_id 그룹화). 예외 타입: `OrganizationNotFoundError`, `OrganizationHasChildrenError`, `DuplicateOrganizationNameError`, `OrganizationInvalidMoveError`.

#### 이메일 발송 (`app/email/`)

- `app/email/transport/base.py` — `EmailTransport` 추상 클래스. `send(EmailMessage) -> None` 단일 추상 메서드. 표준 라이브러리만 사용해 외부 의존성 없음.
- `app/email/transport/m365_oauth.py` — `M365OAuthSmtpTransport`. Azure AD `msal.ConfidentialClientApplication` 으로 XOAUTH2 액세스 토큰 발급 → `smtplib.SMTP("smtp.office365.com", 587)` STARTTLS → 명시적 `smtp.ehlo()` 재전송 → `AUTH XOAUTH2`. STARTTLS 직후 EHLO 를 명시적으로 재전송하는 이유: RFC 3207 §4.2 요구사항이며, Python `smtplib.starttls()` 는 내부 상태(`ehlo_resp`/`helo_resp`)를 None 으로 클리어만 하고 EHLO 를 자동 재전송하지 않는다. `raw docmd()` 도 `ehlo_or_helo_if_needed()` 를 호출하지 않아 명시적 호출 없이는 M365 가 `503 5.5.2 Send hello first` 를 반환한다. 매 `send()` 마다 msal 인스턴스 새로 생성(토큰 캐시 포기, 항상 최신 자격증명 보장).
- `app/email/transport/factory.py` — `build_transport(session)`. `SystemSetting["email.transport.type"]` 값(`m365_oauth`)에 따라 구현체를 인스턴스화. 향후 옵션 추가 시 `constants.ALLOWED_EMAIL_TRANSPORT_TYPES`에 값 추가 + `elif` 분기 1줄로 확장 완결.
- `app/email/config.py` — `M365OAuthSmtpConfig` frozen dataclass + `load_m365_oauth_config(session)`. `SystemSetting` 5개 키(tenant_id/client_id/client_secret/sender_address/from_display_name)를 읽어 반환. 미설정 키는 `constants.DEFAULT_*` fallback.
- `app/email/message_builder.py` — 두 종류의 빌더를 노출한다. ① `build_plain_text_message(to, subject, body)` — 기존 plain text 전용 (Phase A-1 테스트 발송용). ② `build_multipart_message(recipient, subject, text_body, html_body, sender_address, sender_display_name)` — text/plain + text/html 두 alternative 를 가진 multipart/alternative EmailMessage (Phase A-2 Part 2 공고 포워딩용). 부속 헬퍼: `build_default_forward_subject`, `build_forward_text_body(sender_organizations: list[Any], ...)`, `build_forward_html_body(sender_organizations: list[Any], ...)`, `build_announcement_detail_url`. `_build_sender_info_text_block` / `_build_sender_info_html_block` 가 발신자 정보(사용자 ID·조직명 목록·이메일) 표를 각각 plain text 줄 블록과 인라인 CSS HTML 표로 생성해 본문 메타 박스 바로 아래에 삽입한다. ③ Phase A-3 단체 Daily Report 용 `build_daily_report_subject` / `build_daily_report_text_body` / `build_daily_report_html_body` — forward 빌더의 grayscale 인라인 CSS·헤더 박스·footer 패턴을 재사용하고 5종 카테고리(신규·내용변경·접수예정·접수중·마감 전이) 섹션을 빈 카테고리 생략 + 카테고리당 50건 cap 으로 렌더한다. **[00136] Daily Report 본문 컨테이너 `max-width` 를 `600px` → `~1160px` 로 확장. 각 공고 항목 렌더링을 `app/rendering/announcement_row` 공유 렌더러 호출로 전환 — 출처 배지·상태/전이·공고명·접수/마감 일시가 대시보드 expand 행과 동일한 포맷으로 표시됨. [00139] 각 카테고리 섹션을 단일 셀 `<table>/<td>` 로 감싸 `border:1px solid #e0e0e0;border-radius:6px` 옅은 테두리 박스로 시각적 구분.**
- `app/email/forwarding.py` — 포워딩 로직을 `forward_announcement_prepare` (검증 + `EmailForwardLog` 선 commit + `ForwardPreparation` DTO 반환) 와 `forward_announcement_run` (수신자별 발송 루프 + 최종 status commit) 두 단계로 분리 (00120 비동기화). `ForwardPreparation` 는 두 단계를 잇는 DTO 로 `forward_log_id`, 수신자 목록, 제목, 발신자 헤더, 본문을 담는다. 라우터가 prepare 까지 동기 수행 후 즉시 `forward_log_id` 를 응답하고, run 은 `BackgroundTasks` 로 위임해 별도 `session_scope()` 에서 실행한다. 수신자 1명씩 개별 발송(프라이버시 + per-recipient 추적). 발신자 정보 표에는 사용자가 선택한 단일 발신 조직만 노출. 발신 모듈: `app/email/message_builder` 에 본문 빌드 위임, `app/email/sender.send_with_retry` 에 재시도·이력 위임.

- `app/email/constants.py` — `SystemSetting` 키 상수(`email.transport.type`, `email.m365.*` 5개, `email.from_display_name`, `email.max_retry_count`, `email.send_enabled`, `app.public_base_url`) + 기본값 상수 + `ALLOWED_EMAIL_TRANSPORT_TYPES` frozenset. `email.daily_report.*` 4개 키(`enabled`/`cron_expression`/`last_sent_at`/`test_recipient`)는 단체 Daily Report 설정 영속용으로 `DAILY_REPORT_SETTING_KEYS`/`DAILY_REPORT_SETTING_DEFAULTS` 로 묶여 「메일 설정」 카드와 별개로 관리된다. `RELATED_KIND_FORWARD="forward"` / `RELATED_KIND_TEST_SEND="test_send"` / `RELATED_KIND_DAILY_REPORT="daily_report"` 는 `EmailSendRun.related_kind` 컬럼 값 도메인 상수. `SETTING_KEY_APP_PUBLIC_BASE_URL` 은 메일 본문 URL prefix — email.* 네임스페이스 밖이지만 이메일 본문 전용 용도라 동일 모듈에 배치.

### 데이터 흐름 (delta + snapshot 파이프라인)

1. CLI 가 `init_db()` (멱등) → `load_sources_config()` → enabled 소스 목록 결정.
2. 각 소스마다 `get_adapter()` → `adapter.scrape_list()`. 공고 1건 수집 시: 첨부 즉시 다운로드 → `delta_announcements` / `delta_attachments` 에 INSERT. **본 테이블 직접 UPSERT 안 함**.
3. ScrapeRun 정상 종료 시 단일 트랜잭션: delta → 본 테이블 4-branch UPSERT (변경 감지 + 사용자 라벨링 reset 포함) + KST 날짜별 `scrape_snapshots` 생성/UPSERT (`merge_snapshot_payload`) + delta 비우기. 트랜잭션 실패 시 rollback — delta 는 유지되어 다음 수집에서 재시도 가능.
4. SIGTERM / 실패 중단 시: delta 비우고 본 테이블·snapshot 변경 없음. ScrapeRun.status = `cancelled` / `failed`.
5. 웹 UI 는 본 테이블 조회. viewer 템플릿은 `source_type` 별로 분기.
6. 고아 첨부파일 GC: `scripts/python/gc_orphan_attachments.py` (--dry-run) 수동 또는 APScheduler 일 1회 자동.

### 도메인 모델

- `Announcement` 1:N `Attachment`. 공고 UPSERT 키 = `(source_type, source_announcement_id)` 복합 UNIQUE. `is_current=True` 인 row 가 현재 버전. 상태는 `AnnouncementStatus` Enum (`접수중/접수예정/마감`, 한글 원문 값).
- Attachment 는 `(announcement_id, original_filename)` 복합 UNIQUE + `sha256` 중복 체크.
- **Canonical 레이어**: `CanonicalProject` 별도 테이블이 `Announcement` 1:N 을 FK 로 묶음. `canonical_key` UNIQUE (예: `official:산업통상부공고제2026-300호::2026년도제조암묵지기반AI모델개발사업신규지원대상과제공고`). 각 Announcement 는 `canonical_project_id` FK 로 참조. IRIS 공식 식별자는 `ancmNo`. official 키는 ancmNo + 공고명 합성 (false-positive 분리). 분석은 `docs/canonical_identity_design.md` + `docs/duplicate_detection_analysis.md`.
- **증분 UPSERT 4-branch** (delta apply 단계에서 수행):
  - (a) `created` — 같은 (source_type, source_id) is_current row 없음 → INSERT.
  - (b) `unchanged` — 비교 4 필드 동일 → no-op.
  - (c) `status_transitioned` — status 만 변경 → in-place UPDATE.
  - (d) `new_version` — 그 외 변경 또는 첨부 sha256 변경(2차 감지) → 기존 row 봉인(is_current=False) + 신규 row INSERT + 사용자 라벨링 reset.
  - 비교 필드: `title, status, deadline_at, agency`. `received_at` 은 접수예정 단계의 미기재→보완 패턴이 잦아 의도적으로 제외. `_upsert_announcement` 는 비교 필드 파라미터화로 1차(목록 필드) / 2차(첨부 sha256) 양쪽에서 재사용.
- **변경 시 리셋** (status 단독 전이 제외): 공고 내용 변경 감지 시 `AnnouncementUserState.is_read` 전부 False 재설정, `RelevanceJudgment` 를 `RelevanceJudgmentHistory` 로 이관(`archive_reason="content_changed"`) 후 삭제, `FavoriteEntry` 유지. delta apply 와 동일 트랜잭션이라 apply 실패 시 rollback 보장.
- **Backfill / 검증 스크립트**: `scripts/python/backfill_canonical.py` (구버전 IRIS 데이터 재계산, 멱등), `verify_canonical_iris.py` (IRIS 단독 중복 케이스), `verify_canonical_cross_source.py` (IRIS↔NTIS cross-source false positive/negative).

### 실행 환경

Docker Compose. `app` (FastAPI UI) · `scraper` (1회성 배치, `profiles: [scrape]`) 가 같은 이미지를 공유. 호스트 Python 직접 실행은 지원 안 함.

스크래퍼 실행 경로 **3 가지** — (1) 호스트 쉘에서 `docker compose --profile scrape run --rm scraper` (CLI), (2) 웹 UI `/admin/scrape` "지금 시작" 버튼이 app 컨테이너 안에서 호스트 dockerd 를 subprocess 로 조작, (3) APScheduler cron / interval 잡. **세 경로 모두 같은 `scrape_runs.status='running'` row 1개로 lock**.

- **바인드 마운트**: `./data` (DB·다운로드·로그·백업), `./app` (소스 코드), `./sources.yaml:/run/config/sources.yaml` (app 은 `:ro` 제거 — 편집기 쓰기 필요, scraper 는 `:ro` 유지), `/var/run/docker.sock` (**app 서비스만** — 웹이 subprocess 로 docker CLI 호출), `./scripts:/app/scripts` (운영자 CLI), `/etc/localtime:/etc/localtime:ro` (호스트와 컨테이너 시각 일치, slim 이미지의 tzdata 미탑재 보완), `./.env:${HOST_PROJECT_DIR}/.env:ro` (compose subprocess 가 `--project-directory ${HOST_PROJECT_DIR}` 기준으로 `.env` 탐색 — 타겟의 `${HOST_PROJECT_DIR}` 보간은 `run_compose.sh` 가 `--project-directory "$SCRIPT_DIR"` 를 명시해 항상 프로젝트 루트 `.env` 에서 읽도록 결정론적으로 고정됨(00134)), `./alembic:/app/alembic:ro` · `./alembic.ini:/app/alembic.ini:ro` (이미지 COPY 보다 우선 적용 — 신규 migration 추가 시 이미지 재빌드 불필요).
- **docker-in-docker 아님**: app 컨테이너는 자체 dockerd 를 띄우지 않고 호스트 `/var/run/docker.sock` 으로 호스트 dockerd 를 원격 조작. 이미지에 `docker.io` + `docker-cli` (apt — 두 패키지 분리) + compose v2 플러그인 (`docker-compose-linux-<arch>`) + `docker-compose.yml` 자체를 COPY. `docker compose -f /app/docker-compose.yml --project-directory $HOST_PROJECT_DIR ...` 형태로 호출. app 서비스의 `HOST_PROJECT_DIR` 는 `env_file: - .env` 단일 출처로 주입(00134 — `environment` 블록 중복 정의 시 보간 실패로 빈 값이 `env_file` 정상값을 덮어쓰는 간헐적 버그가 있어 제거). `HOST_PROJECT_DIR` 미설정 시 `ComposeEnvironmentError` 로 ScrapeRun failed.
- **호스트 docker 그룹**: docker.sock 접근을 위해 컨테이너 내부 유저가 호스트 `docker` 그룹 gid 를 보조 그룹으로 가져야 한다. `HOST_DOCKER_GID` (`.env`) → `docker-compose.yml` `group_add` 로 전달. gid 불일치 시 "Permission denied on docker.sock".
- **호스트 UID/GID 매핑**: 두 서비스에 `user: "${HOST_UID}:${HOST_GID}"`. `.env` 명시 설정 필수 (자동 fallback 없음). 컨테이너가 만드는 파일 owner 가 호스트 유저가 됨. `docker-compose.yml` `build.args` 가 `HOST_UID`/`HOST_GID` 를 Dockerfile 의 `useradd` 로 전달해 `/etc/passwd` 엔트리를 생성 — UID 변경 시 `docker compose build` 재실행 없이는 "I have no name!" 재발 (entrypoint 가 동적 패치 불가).
- **Playwright 브라우저 경로**: `PLAYWRIGHT_BROWSERS_PATH=/opt/playwright` 공용 경로. 빌드 시 `chmod -R a+rX` 적용해 비특권 UID 기동에서도 chromium 접근 가능.
- **subprocess 규약**: `subprocess.Popen(..., start_new_session=True)` 새 프로세스 그룹(`os.setsid`) → 중단은 `os.killpg(pgid, SIGTERM)` 로 그룹 전체 전파 → compose v2 가 관리 컨테이너 PID 1 로 릴레이. detach 금지. stdout/stderr 는 **파일 redirect** (PIPE 는 버퍼 가득 차면 자식이 블로킹).
- **entrypoint 설정 주입**: `docker/entrypoint.sh` 가 마운트된 `sources.yaml` 을 per-run 임시 디렉터리(`mktemp`)로 복사하고 `SOURCES_CONFIG_PATH` 환경변수를 주입한 뒤 `python -m app.cli` exec. **sources.yaml 우선순위**: (1) bind mount → (2) 이미지 내 `sources.yaml.template` 폴백. 웹/스케줄러가 "이번 run 만 특정 소스" 로 제한할 때는 `SCRAPE_ACTIVE_SOURCES` (쉼표 연결 문자열) 환경변수로 전달 — 파일을 수정하지 않고 per-run 복사본 격리를 유지.
- **동시성 lock**: `create_scrape_run` 이 단일 트랜잭션 안에서 SELECT(running 존재 검사) → INSERT 해 race 방지. running row 가 곧 "실행 중 flag". 웹 startup 의 `cleanup_stale_running_runs()` 가 pid 없는 / 사라진 running row 를 failed 로 정리해 lock 자동 해제 → 이후 `start_scheduler()`.
- **스케줄러 단일 인스턴스 (00131)**: cron 중복 실행 버그의 근본 원인 — OS crontab 이 아니라 웹 프로세스 내부 APScheduler `BackgroundScheduler` 의 다중 인스턴스다. uvicorn `--reload` 가 항상 켜져 있어 코드 변경/재기동마다 worker 프로세스가 교체되는데, 이전 worker 의 `BackgroundScheduler` 가 깨끗이 정리되지 못하고 살아남으면 같은 `scheduler_jobs` 테이블을 보는 스케줄러가 N개가 되어 동일 job 이 trigger 시각마다 N회 실행된다(`scheduler_jobs` row 중복이 아니라 멀티-인스턴스 — '2번→3번' 점증 증상과 일치). 대책: `app/scheduler/single_instance.py` 의 고정 경로 flock(`<DB 디렉터리>/scheduler-single-instance.lock`) 으로 '컨테이너당 job 을 실행하는 스케줄러는 1개' 불변식을 강제 — lock 획득에 성공한 프로세스만 `start_scheduler()` 가 스케줄러를 띄우고, 실패한 프로세스는 띄우지 않는다. `stop_scheduler()` 가 정지와 함께 lock 을 해제해 다음 worker 가 승계한다.
- **스케줄 job single-flight 가드 (00131-2)**: 위 단일 인스턴스 flock 이 어떤 이유로 실패하거나 좀비 worker 가 잠시 겹치는 구간을 대비한 defense-in-depth. `app/scheduler/job_guard.py` 의 `try_claim_schedule_slot(job_name)` 이 `scheduler_job_claims` 테이블에 `(job_name, slot_key)` UNIQUE row 를 INSERT 한다 — `slot_key` 는 현재 시각을 60초 시간창으로 내림한 버킷 문자열이라, 동일 trigger 시각에 거의 동시에 발사된 2~3개 호출이 같은 키를 계산하고 **먼저 INSERT 한 단 한 호출만** claim 을 얻는다(나머지는 IntegrityError → WARN 로그 후 skip). `scheduled_scrape`/`scheduled_daily_report_job`/`scheduled_backup_job`/`gc_orphan_attachments_job` 4개 잡 함수가 진입부에서 호출 — 메일 발송·scrape run 생성·백업 부수효과가 동일 주기에 1회만 일어난다. claim 트랜잭션은 본 작업 트랜잭션과 분리해 먼저 commit(00128 의 SQLite 단일 writer 충돌 교훈). UNIQUE 위반이 아닌 예외는 fail-open(가드 버그로 정기 작업이 멈추지 않게). claim row 는 7일 보관 후 자동 정리.
- **스케줄러 ASGI startup 훅 기동 (00133)**: task 131 회귀 수정. `app = create_app()` 은 모듈 수준 코드라 import 마다 실행되는데, uvicorn `--reload` 환경에서는 reload 감시 부모 프로세스(컨테이너 pid 1)와 실제 요청을 처리하는 worker 프로세스가 **둘 다** `app.web.main` 을 import 한다. 예전엔 `create_app()` 본문에서 직접 `start_scheduler()` 를 호출해, 00131 의 단일 인스턴스 flock 을 부모 프로세스가 worker 보다 먼저 선점 → worker 스케줄러가 영구 미기동 → [스케줄]/[시스템 백업] 탭이 'APScheduler 가 기동되지 않았습니다' 경고를 계속 표시했다(컨테이너 로그상 pid 1 은 lock 획득·기동 성공, worker pid 는 lock 획득 실패). 대책: 스케줄러 기동(`start_scheduler` + `ensure_backup_cron_registered` + `ensure_daily_report_cron_registered`)을 FastAPI `on_event("startup")` 훅(`_startup_scheduler`)으로 이동. startup 훅은 ASGI 서버가 실제로 가동되는 프로세스(=worker)에서만 실행되므로, flock 을 쥐는 프로세스와 HTTP 를 처리하는 프로세스가 항상 일치한다. 00131 의 flock 단일 인스턴스 가드는 그대로 유지(멀티 worker / reload 중 worker 겹침 시 여전히 1개만 기동).
- **중단 시 atomic 보장**: SIGTERM handler 가 `_cancel_flag` 만 세움. break 는 공고 / 소스 루프 시작 지점에서만. 현재 처리 중 공고는 끝까지 마무리. ScrapeRun.status 도메인: `running/completed/cancelled/failed/partial`.

## 컨벤션

- **언어 / 문서**: 모든 모듈 docstring 과 주석은 **한국어**. 외부 공개용이 아니므로 격식보다 명확성 우선.
- **타입힌트**: 모든 공개 함수에 타입 어노테이션. `from __future__ import annotations` 를 파일 상단에.
- **설정 접근**: 직접 `os.environ` 읽지 말고 `app.config.get_settings()` 싱글턴 경유.
- **시각 처리**: DB 의 모든 시간 컬럼은 timezone-aware UTC 저장이 컨벤션. **SQLite 백엔드는 `DateTime(timezone=True)` 의 tz 를 실제로 저장하지 못해** SELECT 결과가 naive 로 돌아오므로, 비교 직전에 양쪽을 UTC tz-aware 로 정규화하는 헬퍼 (`_as_utc`) 를 사용한다. **표시·로그·cron 은 Asia/Seoul (KST) 단일 고정**: 저장은 `now_utc()`, 표시는 `to_kst()` + `format_kst()` 또는 Jinja2 필터 (`kst_format`, `kst_date`). **시스템 표시 포맷은 `YYYY-MM-DD HH:mm:ss` 초 단위로 통일** (`DEFAULT_KST_FORMAT = "%Y-%m-%d %H:%M:%S"`). `kst_date` Jinja2 필터도 초 단위 표시 (날짜 전용 아님 — 00122). JS datetime 은 `en-CA` locale + `Asia/Seoul` timeZone 패턴으로 통일 (`toLocaleString('ko-KR')` 대체). `datetime.utcnow()` · naive `datetime.now()` 직접 사용 금지 — `app/timezone.py` 헬퍼로만 생성. 컨테이너 `TZ` env 는 유지하되 코드는 `ZoneInfo("Asia/Seoul")` 명시로 환경변수 비의존. 외부 응답(IRIS / NTIS) 날짜는 KST 가정 후 UTC 변환 저장 (가정이 틀렸을 때는 `raw_metadata` 의 원문 텍스트로 backfill). **API 응답 datetime 직렬화**: SQLite SELECT 가 돌려주는 naive datetime 을 `.isoformat()` 하면 `+00:00` 오프셋이 없는 문자열이 나와 프론트엔드 `new Date()` 가 브라우저 로컬 시각으로 해석 — KST 변환이 무력화된다. API 응답에서 datetime 을 직렬화할 때는 `_as_utc(dt)` 로 UTC tz-aware 로 정규화한 뒤 `.isoformat()` 해야 한다 (결과에 `+00:00` 포함). JS 측 `formatDateTimeKst` / `formatDateKst` 는 오프셋 있는 입력을 전제로 동작한다.
- **UPSERT**: 공고는 `(source_type, source_announcement_id)` 복합 키, 첨부는 `(announcement_id, original_filename)` + `sha256` 비교. 재실행 시 중복 생성 금지. 변경된 경우만 신규 row 생성(이력 보존), 상태 전이만은 in-place UPDATE.
- **예외 격리**: 스크래퍼 파이프라인은 **공고 1건 단위** try/except. 한 공고 실패가 전체 실행을 중단시키지 않는다.
- **웹 보안 경계**: FastAPI 외부 노출 금지 (Secure=False 쿠키, CSRF 토큰 없음 — 로컬 전용 전제). 첨부 다운로드는 반드시 `download_dir` 하위로 경로 트래버설 방어. POST 인증 라우트는 `ensure_same_origin` 의존성 (Origin/Referer 헤더 netloc 비교).
- **settings 라우트의 세션 재조회**: `current_user` 는 `_auth_db_session` 소속 인스턴스다. email·notification·password 변경처럼 mutate 목적으로 `User` 가 필요한 settings 라우트에서는 반드시 해당 라우트의 세션 기준으로 재조회 (`session.get(User, current_user.id)`) 한다. 다른 세션에서 로드된 인스턴스는 ORM identity map 이 추적하지 않아 flush/commit 해도 UPDATE 가 발행되지 않는다.
- **로깅**: loguru + stdlib 브리지 (`InterceptHandler`). 레벨은 `LOG_LEVEL` 환경변수 (`DEBUG/INFO/WARNING/ERROR/CRITICAL`). 웹 프로세스는 `create_app()` 최상단에서 `configure_logging()` 1회 호출 필수. uvicorn / starlette 계열은 자체 StreamHandler 제거하고 propagate=True 로 전환해 중복 출력 방지. sqlalchemy / alembic 은 DEBUG 시에도 최소 INFO 고정. 모든 요청·응답은 `LoggingMiddleware`, 미처리 예외는 전역 핸들러가 stack trace 포함 강제 출력.
- **패키지 경로**: uvicorn 경로 `app.web.main:app`, 스크래퍼 CLI `python -m app.cli` (서브커맨드 없음, CLI 인자 없음). 재구조화 시 `docker/entrypoint.sh` 와 compose `command` 도 함께 조정.
- **Jinja2 `<script>` 블록 내 태그 리터럴 금지**: Jinja2 파서는 `<script>` 안의 JavaScript 주석까지 포함해 `{% %}` 태그를 해석한다. script 블록 주석에서 Jinja 태그 문자열을 언급해야 하면 `{% else %}` 같은 리터럴 대신 "else 분기" 등 플레인 텍스트로 대체한다 — 위반 시 `TemplateSyntaxError` 로 페이지 전체 500.
- **Jinja2 템플릿에서 Python 빌트인 생성자 호출 금지**: `set()`, `dict()`, `tuple()` 등을 Jinja2 표현식에서 호출하면 렌더링 500. 빈 컬렉션 기본값은 리터럴 `[]` / `{}` 로.
- **로컬 Python 직접 실행 금지**: 호스트에서 `python -m app.cli` 실행 미지원. Docker Compose 경유만.
- **실행 스크립트는 `./run_compose.sh` / `./run_admin.sh`**: 직접 `docker compose` 명령은 사용 금지 — wrapper 가 단일 `docker-compose.yml` 을 항상 사용. 서브커맨드(up/down/build/logs/restart/ps/scrape) 뒤의 인자는 그대로 docker compose 에 전달.
- **lint / format**: ruff (line-length 110, target py311, E501 무시). mypy 는 옵션 dev 의존성으로만.
- **N+1 제거 패턴 (목록 페이지)**: 로그인 사용자 종속 데이터(읽음·관련성·즐겨찾기·동일과제) 는 페이지의 announcement_id / canonical_id 전체를 `IN` 절로 한 번에 SELECT 해 dict 반환하는 `get_X_map(ids)` 헬퍼로 일괄 조회 후 템플릿에 주입. 목록당 추가 쿼리는 2~3개로 고정. 비로그인은 헬퍼 호출 자체를 skip — 기본 경로 오버헤드 0. 폴더 트리처럼 폴더 수가 가변적이고 `IN` 절보다 GROUP BY 가 더 적합한 경우 (즐겨찾기 사이드바 폴더별 카운트 `count_favorite_entries_by_folder_for_user`) 는 JOIN + GROUP BY 단일 쿼리. 관련성 조회는 `get_relevance_summary_by_canonical_id_map(user_id, canonical_ids)` 헬퍼로 본인 소속 조직 판정 + 전체 카운터(✅/❌ 분리) 를 한 번의 쿼리 묶음으로 반환. 비로그인은 카운터만 GROUP BY 단일 쿼리. 페이지당 추가 쿼리 1~2개 고정.

## 주요 결정

각 항목은 "왜 이 결정을 했는가" 를 운영·개발에 필요한 만큼만 기록한다. 동일 주제로 결정이 여러 번 갱신된 경우 식별을 위해 task ID 를 일부 유지한다.

### 데이터 모델 / 도메인

- **상태값을 한글 원문 그대로 Enum 값으로 보존**: IRIS 노출 텍스트와 1:1 매칭되어 필터·표시 변환 비용을 줄이기 위함.
- **증분 수집 — DB 초기화 금지**: 재실행 시 기존 DB 유지하고 변경된 공고만 갱신. `init_db()` 는 Alembic stamp/upgrade 로 스키마만 보장하고 데이터를 절대 건드리지 않는다. 이력 보존 + 불필요한 상세 재수집 방지가 목적.
- **`received_at` 을 변경 감지 비교 필드에서 제외**: 접수예정 상태 수집 시 공란이었다가 이후 보완되는 패턴이 잦아 잡음 트리거를 피하기 위함. 핵심 비교 필드 = `title, status, deadline_at, agency` 4개로 고정.
- **2차 변경 감지 도입**: 1차 감지 (목록 UPSERT 시) 는 sha256 을 모르므로 첨부 변경 탐지 불가. 첨부 다운로드 완료 후 2차 감지를 추가해 첨부 개수·sha256 변동을 감지한다. `_upsert_announcement` 를 비교 필드 파라미터화해 두 단계가 동일 함수를 재사용.
- **사용자 라벨링 데이터 보호 (변경 감지 리셋 정책의 근거)**: 공고 내용이 실질 변경되면 `AnnouncementUserState(is_read)` 와 `RelevanceJudgment` 이력을 재설정 — 사용자 판단이 변경 전 내용 기준으로 오염되지 않도록. `FavoriteEntry` 는 내용 변경과 무관하게 유지.
- **관련성 단위는 canonical, 읽음 단위는 announcement — 혼용 금지**: `RelevanceJudgment` 의 UNIQUE 키는 `(canonical_project_id, user_id, organization_id)` — 조직 판정만 허용 (organization_id NOT NULL, 개인 판정 완전 제거). 같은 canonical 의 여러 소스 공고가 판정을 공유. `AnnouncementUserState.is_read` 는 announcement 단위. 목록 row 는 announcement 이지만 관련성 배지는 canonical_id 로 조회·표시한다.
- **organization_id 는 판정 주체 조직 — NOT NULL 필수**: 개인 판정(organization_id=NULL) 기능은 완전 제거되었고, 모든 `RelevanceJudgment` row 는 반드시 조직을 지정해야 한다. 안 2 (조직별 1개 row partial UNIQUE) 대신 **안 1 (단일 UNIQUE)** 채택 — SQLite·Postgres 양쪽 호환을 단순하게 보장하고 dialect 차이 회피. 같은 조직 안 여러 사용자가 각자 row 를 가질 수 있으며, 의견 분기가 그대로 데이터에 표현된다.
- **관련성 판정 변경은 원자적 이관 (덮어쓰기 금지)**: 기존 판정이 있을 때 새 판정을 저장하면 기존을 `RelevanceJudgmentHistory` 로 이관 (`archive_reason="user_changed"`) 한 뒤 신규 판정 INSERT — 동일 트랜잭션. 직접 UPDATE 로 덮으면 이력이 소실되므로 설계상 금지. `content_changed` 이관과 동일 패턴. `RelevanceJudgmentHistory` 에도 organization_id 컬럼이 동일하게 존재해 이관 시 소속 정보 보존.
- **관련성 목록 셀 — 조직 판정 배지**: 본인이 만든 조직 row 중 가장 최근 것을 큰 배지로 표시. 카운터(✅ N ❌ M)는 본인 조직 판정 + 타인 포함 전체 row 를 집계 — ❓(미검토) 카운터는 제거됨. 비로그인도 카운터·작성자·조직명 모두 동일하게 노출 — 로그인 사용자와 동일 노출 컨벤션. hover 툴팁은 본인 조직 판정(mine_organization) 또는 다른 사용자 판정(others) 중 하나라도 있으면 노출.
- **즐겨찾기 항목은 announcement 단위** (이전 canonical 단위 설계 폐기): 별을 누른 바로 그 공고가 아닌 다른 공고가 대표로 등록되거나, 동일 과제의 다른 공고가 즐겨찾기 목록에 보이지 않는 문제로 폐기됨. 이제 별은 announcement 단위로 1건 저장되고, "동일 과제 모두 저장" 라디오는 같은 canonical 그룹의 is_current 공고 전체를 한 번에 등록하는 수단으로 동작한다.
- **FavoriteFolder depth 2 제약은 ORM validator**: SQLite 의 ALTER TABLE 제약 추가가 어렵고 단순 계층 구조라서 DB CHECK 대신 ORM `@validates` 레벨에서 depth 검증.
- **SQLite UNIQUE(parent_id IS NULL) 루트 동명 중복은 app-level SELECT 체크**: SQLite·Postgres 모두 UNIQUE 제약에서 NULL 을 "서로 다름" 으로 취급해 parent_id=NULL 루트 폴더의 동명 INSERT 가 DB 제약으로 막히지 않는다. INSERT 전 SELECT 로 직접 중복 확인하는 app-level 체크 패턴 적용. 루트 조직 동명 방지도 동일 패턴.

### Canonical 키 (`compute_canonical_key`)

- **`compute_canonical_key` 공식 키 우선 + fuzzy fallback**: official 키 (ancmNo + 공고명 합성) 를 우선, 없으면 (제목 불용어 제거 + 주관기관 정규화 + 마감연도) 조합 fuzzy. NTIS cross-source 실매칭에서도 fuzzy → official 승급 경로가 검증됨. 정규화 규칙 상세는 `docs/canonical_identity_design.md §4-1 / §11-6`.
- **official canonical_key 합성 키 도입 (00039)**: 공고번호 단독 official 키에서 동일 ancmNo 아래 세부 공고가 다른 false-positive 가 발생해 포맷을 `"official:{normalized_ancmNo}::{normalized_title}"` 로 변경. title 정규화 = NFKC → NTIS suffix 제거 (`\s*_\([0-9]{4}\).*$`) → 공백 전체 제거. 구분자 `::` 는 ancmNo 가 단일 콜론을 포함하지 않는다는 가정 위에서 사용. fuzzy scheme 은 변경 없음. 분석은 `docs/duplicate_detection_analysis.md`, 키 포맷·검증은 `docs/canonical_identity_design.md §11-6`.
- **IRIS 공식 식별자는 `ancmNo`**: 목록 API 전수 필드에서 비어 있는 레코드 0건. 1:1 고유 키가 아니라 "공고 묶음" 단위 (하나의 공고번호에 여러 세부과제 ancmId 가 대응). canonical_key 생성 전 NFKC 정규화 + 공백 제거 필수. 탐사 결과는 `docs/canonical_identity_design.md §2`.
- **NTIS 는 httpx 단독 수집**: 목록·상세·첨부 모두 httpx POST SSR. IRIS 와 달리 AJAX API 없음. 첨부도 `wfUid + roTextUid` 파라미터로 httpx POST 직접 다운로드 (JS onclick 우회 불필요). `AttachmentDownloader` 는 소스를 모르고 adapter 계층이 분기.

### 수집 파이프라인

- **수집 파이프라인 — delta 기반**: 수집 중 본 테이블 직접 UPSERT 하지 않고 delta 테이블에 INSERT, ScrapeRun 종료 시점에 단일 트랜잭션으로 (delta → 본 테이블 4-branch + snapshot UPSERT + delta 비우기) 일괄 적용. 부분 실패 시 본 테이블·snapshot 변화 없음 보장. 첨부 파일 자체는 FS 라 트랜잭션 밖 — 고아 파일은 GC 스크립트로 정리.
- **ScrapeSnapshot — KST 날짜 1 row UPSERT, payload 머지**: `scrape_snapshots.snapshot_date` UNIQUE 1 row. 같은 날 여러 ScrapeRun 결과는 `merge_snapshot_payload(existing, new)` 로 합산. 5종 카테고리: `new` · `content_changed` · `transitioned_to_{접수예정,접수중,마감}`. 머지 규칙: `new`·`content_changed` 는 ID set union; `transitioned_to_X` 는 동일 id 마지막 to 갱신 + from==to 면 제거 (실질 변화 없음). payload 는 JSON 범용 컬럼.
- **고아 첨부파일 GC**: 첨부 다운로드는 FS 라 트랜잭션 보호 불가. delta apply 실패 / SIGTERM 시 다운로드된 파일이 DB 없이 남을 수 있다. `scripts/python/gc_orphan_attachments.py` (--dry-run) 가 `data/downloads/` FS vs `attachments.file_path` 대조해 정리. APScheduler 일 1회 또는 수동 실행.
- **날짜 기반 수집 중단 조건 제거**: 정부 공고 사이트가 마감일을 `2200-01-01` 등 허위 값으로 기재하는 사례가 많아 날짜로 중단 여부를 판단하면 오작동 가능성이 크다. `list_scraper` 에서 날짜 기반 break 를 모두 제거하고 `max_pages` / `max_announcements` 상한만으로 수집 제어.
- **페이지·공고 수집 상한 — 2 단계 우선순위**: 코드 기본값 `max_pages=10, max_announcements=200`. 우선순위: `sources.yaml scrape:` 전역 > 소스별 설정 > 코드 default. CLI 인자 단계는 없다 (의도). 보수적 기본값으로 차단 회피.
- **목록·상세 수집은 httpx (HTTP-only), 첨부 수집만 Playwright**: IRIS 목록·상세는 SSR 이라 httpx 만으로 충분. 첨부 링크는 `javascript:` onclick 으로만 노출되어 HTTP 직접 접근 불가 → Playwright headless 필수. 이 구분이 의존성 무게를 줄이고 수집 단계 안정성을 높임.

### 시각 / 스케줄러

- **표시 timezone 은 Asia/Seoul 단일 고정 (사용자별 설정 없음)**: 팀 공용 로컬 단일 운영 전제. `app/timezone.py` 헬퍼와 Jinja2 필터 (`kst_format`, `kst_date`) 가 강제. `datetime.utcnow()` 등 naive 생성은 프로젝트 금지 패턴.
- **APScheduler `BackgroundScheduler(timezone=KST)`**: cron 표현식이 KST 기준으로 해석. `30 9 * * *` = KST 09:30. timezone 변경 후 jobstore 기존 잡의 `next_run_time` 이 자동 재해석되도록 startup 에서 trigger 의 timezone 필드를 KST 로 갱신 + `reschedule_job` 호출.
- **APScheduler JobStore write 는 `session_scope` 밖에서 먼저 수행**: `SQLAlchemyJobStore` 는 동일 SQLite 파일의 `scheduler_jobs` 테이블에 **별도 커넥션**으로 write 한다. `set_setting + flush` 로 이미 write 트랜잭션이 열린 `session_scope` 안에서 `register_*_cron_schedule` 을 호출하면 SQLite 단일 writer 제약("database is locked") 으로 500 + 롤백이 발생한다. 해결 패턴: `register_*_cron_schedule` 호출을 `session_scope` **전에** 배치한다. Daily Report 설정 저장(`put_daily_report_settings`)과 백업 설정 저장 모두 동일 패턴 — 코드 내 주석('순서 주의') 참조.
- **외부 응답 (IRIS / NTIS) 날짜는 KST 가정 → UTC 변환 저장**: 소스가 timezone 을 명시하지 않지만 KST 기준으로 게시된다는 관찰. 원문은 `raw_metadata` 에 보존 — 가정이 틀렸을 때 backfill 가능 (`scripts/python/backfill_kst_assumption.py`).
- **APScheduler 3.x 를 웹 프로세스 내부 BackgroundScheduler 로**: 별도 worker 프로세스를 띄우지 않고 FastAPI 와 같은 프로세스 안에서 가동 — 운영 단순성이 팀 공용 로컬 배포 목적에 부합. JobStore 는 `SQLAlchemyJobStore` 단일 (`scheduler_jobs` 테이블, APScheduler 가 자체 생성해 Alembic 관리 밖). 재기동 시 잡 자동 복원. `misfire_grace_time=300` · `coalesce=True` · `max_instances=1` 로 놓친 잡 합치기 + 중복 기동 방지. 4.x 는 API breaking change 라 `apscheduler>=3.10,<4.0` 으로 고정.
- **APScheduler 백그라운드 컨텍스트에서 docker 바이너리 절대경로 탐색**: `scheduled_scrape` 가 호출하는 `build_compose_command` 가 `argv[0]` 을 `"docker"` 문자열로 두면 APScheduler 의 제한된 PATH 에서 `FileNotFoundError`. `_resolve_docker_binary()` 가 `shutil.which` → 관행 경로 (`/usr/bin/docker`, `/usr/local/bin/docker`) → `DOCKER_BINARY` env 순으로 탐색해 절대경로 반환. 탐색 실패 시 `ComposeEnvironmentError` 로 ScrapeRun failed.

### 인증 / 보안

- **FastAPI 로컬 전용 + 세션 쿠키 인증**: "개인용 → 팀 공용" 로드맵으로 auth 도입. **비로그인 열람 유지** — 목록·상세 URL 그대로, `current_user_optional` None 이면 사용자 종속 로직만 skip. 자유 회원가입 (관리자 승인·이메일 검증 X), 비밀번호 bcrypt (passlib, rounds=12), 서버 사이드 세션(`UserSession` row) + HttpOnly / SameSite=Lax 쿠키 (Secure=False, 로컬 HTTP 전제), 세션 ID `secrets.token_urlsafe(32)` 평문 DB 저장. JWT 는 의도적으로 선택하지 않았다 (운영 복잡도 증가). 정책 상수와 근거는 `app/auth/constants.py`.
- **CSRF 토큰 대신 same-origin 체크**: 로컬 팀 전용 전제라 CSRF 토큰은 과잉. POST 인증 라우트는 `ensure_same_origin` 의존성으로 Origin / Referer 헤더만 netloc 비교. 헤더가 없는 비-브라우저 호출 (curl 등) 은 통과시켜 스크립트 운영 마찰을 줄인다.
- **읽음 / 안읽음 UI 는 로그인 시만 활성**: 비로그인은 헬퍼 호출 자체를 skip — 기본 경로 오버헤드 0. 상세 진입 시 `mark_announcement_read` 로 UPSERT.
- **첫 관리자 계정은 CLI 로만 생성**: 자유 회원가입은 항상 `is_admin=False`. 운영자가 `scripts/python/create_admin.py` (getpass 2회 + argparse username/--email) 로 컨테이너 안에서 1회 생성. 컨테이너 내부 TTY 실행 전제.

### 조직 / 사용자 관리

- **조직은 users FK 컬럼이 아닌 별도 테이블 + M:N 조인**: 사용자가 0개 이상 조직에 속할 수 있어 단일 FK 불가. `organizations` (트리) + `user_organizations` (조인) 두 테이블. `ON DELETE RESTRICT` (자식·멤버 확인 강제), 삭제 시 app-level 체크 선행.
- **조직 트리 조회는 재귀 CTE 대신 메모리 빌드**: 로컬 소규모 데이터 전제로 단순성 우선 — 전체 SELECT 후 parent_id 그룹화로 조립. 데이터가 수만 건 이상으로 커지면 재귀 CTE 전환 고려.
- **조직 import 의 user_organizations 재매핑은 이름 경로 기반**: import 로 조직 ID 가 변경·삭제될 때 기존 user_organizations 행을 그대로 두면 FK 에러 또는 고아 매핑 발생. import 전 `org_id → 이름 경로 (루트부터 name tuple)` 스냅샷을 메모리에 보존 → 기존 organizations·user_organizations 전체 삭제 → 새 트리 INSERT → 이름 경로로 user_organizations 재매핑. 이름 경로가 사라진 매핑은 에러 없이 드롭하고 영향받은 사용자 수·드롭 수를 flash 로 안내.

### 공고 진행 상태 (Progress)

- **선점 제약은 DB UNIQUE 대신 app-level transactional check**: `진행` 또는 `종료` 상태는 한 canonical 당 1 조직만 허용 — 어느 한 조직이 `진행`/`종료` 를 선택하면 다른 조직은 해당 두 상태 선택 불가. SQLite partial unique index 문법 복잡성 + Postgres 이식성을 동시에 회피하기 위해 DB UNIQUE/CHECK 대신 SELECT(진행·종료 row 존재 검사) → INSERT 단일 트랜잭션 패턴으로 보장 (SQLite 단일 writer 가정). 409 Conflict 은 라우터가 `PreemptionConflict` 예외를 캐치해 반환. `관심/검토` 는 조직별 동시 복수 허용.
- **progress 권한 = 조직 멤버 누구나 (relevance 와 의도적 차이)**: 관련성 판정은 row 작성자 본인만 수정·삭제 허용. progress 는 같은 조직 소속이면 작성자와 상관없이 수정·삭제 가능 — 작성자 휴가·퇴사 시 협업 의사결정을 위한 결정. repository 는 권한 검증 미포함, 라우터가 전담.
- **progress UNIQUE 키는 (canonical_project_id, organization_id)**: 동일 조직이 같은 canonical 에 row 를 중복 생성할 수 없다. `created_by_user_id` 는 "마지막 수정자" 메타로만 보존. relevance 의 `(canonical_project_id, user_id, organization_id)` 3-tuple 과 다르다.
- **진행 단계 4종**: `관심 / 검토 / 진행 / 종료`. `진행`·`종료` 가 mutex 선점 제약 (한 canonical 당 두 상태 통합 1 조직), `관심`·`검토` 는 복수 조직 동시 가능. 목록 필터는 없음·진행중·검토중·내조직진행중·내조직검토중 다중 체크박스 OR. 비로그인 시 "내 조직 ..." 옵션은 silent drop. 설계 상세는 `docs/progress_org_design.md`.

### 게시판 (boards.sqlite3)

- **boards.sqlite3 분리 DB**: 메인 `app.sqlite` 리셋이 필요한 상황에서도 게시글이 유실되지 않도록 별도 파일로 분리. 건의사항·공지사항 한 파일을 공유 (테이블 분리). Alembic 관리 밖이며 `Base.metadata.create_all()` 로 자체 초기화.
- **Cross-DB 고아 작성자 방어**: 메인 DB reset 시 `Suggestion.author_id` / `SuggestionComment.author_id` FK 가 소실되어 사용자 정보를 조인할 수 없는 상황 대비. `author_validity.py` 헬퍼가 cross-DB 로 FK 유효성을 검증하고, 유효하지 않으면 작성자 표시를 None (익명) 처리. 정상 서비스는 유지하되 관리자에게만 원문 author_id 노출.
- **boards 테이블 `updated_at` 은 INSERT 시 NULL, 수정 시만 채움**: ORM `onupdate` 콜러블만 채우고 `default` 는 제거. 프런트엔드는 `updated_at IS NOT NULL` 일 때만 "수정됨" 라벨 노출. 기존 운영 DB 는 `ensure_updated_at_initial_null_backfill()` 이 `updated_at = created_at` 행을 NULL 로 backfill (create-copy-drop-rename 패턴, FK 비활성화 후 단일 트랜잭션 + `PRAGMA foreign_keys_check`).
- **boards 게시판 삭제는 소프트 삭제 (`deleted_at`)**: DELETE 대신 `deleted_at = now_utc()` in-place UPDATE. 모든 조회·목록·카운트 쿼리에 `deleted_at IS NULL` 필터. `delete_suggestion` 은 게시글 + 소속 댓글을 ORM cascade 대신 명시적 bulk UPDATE 로 일괄 처리 (SQLAlchemy soft-delete cascade 미지원 우회). 댓글 단독 삭제 (`delete_suggestion_comment`) 도 소프트 삭제로 통일.

### 대시보드

- **Canonical 스키마 — (B) 별도 테이블 선택**: `announcements` 에 컬럼 추가 (A) 보다 `canonical_projects` 별도 테이블 + FK 참조 (N:1) 구조 (B) 선택. 이유: cross-source 그룹 단위 검색·UI 표시 시 JOIN 없이 그룹 메타를 바로 조회 가능 + 그룹 수준 필드 확장 용이.
- **대시보드 캘린더는 자체 구현 (외부 라이브러리 미사용)**: Pikaday 같은 외부 라이브러리 대신 월별 그리드 캘린더를 직접 구현. 외부 CDN 금지 컨벤션상 번들링이 필요한데, snapshot_date 강조·클릭 비활성 등 커스텀 요구사항이 많아 자체 구현이 번들 무게 대비 유리. 트레이드오프는 `docs/dashboard_design.md`.
- **snapshot 변화 0건 수집 날은 캘린더 비활성 — 디자인 의도**: `merge_snapshot_payload` 는 변화 0건이면 `scrape_snapshots` row 를 생성하지 않는다. 캘린더는 `snapshot_date` 존재 여부로 가용성을 판정하므로 "수집은 했지만 변화 없음" 인 날은 비활성 — 의미 없는 기준일 선택을 막는 의도적 설계.
- **대시보드는 완전 read-only, 서버 사전 계산 + JSON 임베드**: 대시보드 내 사용자 액션·POST 엔드포인트 없음. Chart.js 추이 차트 데이터는 서버에서 사전 계산 후 페이지에 JSON 임베드 (별도 API 호출 없음). 컨트롤 변경 시 즉시 페이지 reload — 자동 갱신 없음.

### 공유 렌더링 (`app/rendering/`)

- **이메일·대시보드 공고 행 공유 단위는 view-model + 인라인 CSS HTML 렌더러**: 대시보드는 `style.css` 클래스 기반, 이메일은 인라인 CSS 기반이라 템플릿/CSS 파일 자체는 공유 불가. 공유 가능한 최소 단위를 '공고 1행 데이터(view-model) + 인라인 CSS HTML 조각 렌더러'로 정의하고 `app/rendering/announcement_row.py` 에 단일화. 대시보드는 Jinja global(`render_announcement_row`)을 통해, 메일은 직접 import 해 동일 렌더러를 소비 — 표현 로직을 한 곳에서 관리.
- **공고 행 레이아웃은 `<table>/<td>` — `display:flex + gap` 금지**: Outlook(Word 엔진)·Gmail 이 `flex` 와 `gap` 을 제거하면 항목 사이 간격이 모두 사라진다(메일에서 필드들이 뭉쳐 보이는 원인). `<table>/<td>` + 셀별 명시 `padding` 은 메일 클라이언트가 안정적으로 지원하며, 대시보드(브라우저)에서도 그대로 렌더된다 — 공유 렌더러 한 곳의 변경으로 두 환경을 동시에 수정.
- **배지 색상은 렌더링 모듈이 단일 소스**: `SOURCE_BADGE_COLORS` / `STATUS_BADGE_COLORS` 상수가 `style.css` 의 `.source-iris`·`.status-receiving` 등 클래스 색상과 1:1 연동. 색상을 바꿀 때는 `style.css` 와 이 상수 두 곳을 함께 수정해야 한다.

### 인프라 / 운영

- **SQLite 기본 + `DB_URL` 로 Postgres 전환 가능**: 배포·공유 부담이 없는 단일 파일 DB 가 개인용 목적에 맞음. ORM 은 JSON 범용 타입을 사용해 접속 문자열만 바꾸면 Postgres 전환 가능.
- **DB 백업은 `sqlite3.Connection.backup()` 방식**: `shutil.copy2` 는 진행 중 트랜잭션이 있을 때 손상 위험이 있어 SQLite 공식 hot-backup API 사용. 백업 대상은 app.sqlite + boards.sqlite — 첨부 다운로드 파일은 사용자 원문 명시로 제외.
- **백업 정책은 `SystemSetting` key-value 테이블에 영속**: cron 표현식·최대 보관 수를 설정 파일/환경변수가 아닌 DB 에 저장해 관리자 페이지에서 즉시 반영 가능. 보관 관리는 타임스탬프 그룹 단위로 max_count 개 유지, 초과분 자동 삭제.
- **백업 롤백/복원 기능 미구현**: 수동 복원은 관리자가 보관된 파일을 직접 사용. `BackupHistory` 는 이력 조회 전용(성공/실패·대상·크기·소요시간 기록).
- **Alembic 도입 완료**: `alembic/versions/` 아래 migration 파일로 스키마 관리. `init_db()` 가 Alembic Python API (`alembic.command.upgrade/stamp`) 로 stamp vs upgrade 분기를 자동 선택. 기존 DB 는 `stamp head` 데이터 무변경, 신규 DB 는 `upgrade head`. `env.py` 는 URL 을 `alembic_config` 에서 우선 읽어 호출자가 주입한 엔진을 존중.
- **첨부 바이너리는 DB 밖 FS 에 저장**: hwp / zip 이 수십 MB 가 될 수 있어 BLOB 컬럼은 부적절. DB 엔 경로·해시만. 중복·변경은 `sha256` 으로 판정.
- **Docker `scraper` 서비스를 `profiles: [scrape]` 로 격리**: 스크래퍼는 1회성 배치라 기본 `compose up` 에서 기동되면 안 됨. 명시적 `--profile scrape run --rm` 으로만 실행.
- **외부 CDN 의존 없는 정적 리소스**: 오프라인·격리 환경에서도 UI 가 그대로 동작하도록 CSS·Chart.js 를 `app/web/static/` 패키지 내부에 둔다.
- **sources.yaml 을 gitignore 전환 + template 도입**: sources.yaml 이 git 에 커밋되어 있으면 브랜치 전환 시 로컬 수정본이 롤백되는 문제. `sources.yaml.template` 만 버전 관리. 이미지에도 template 을 COPY 해 두어 마운트 누락 시 폴백.
- **sources.yaml 웹 편집기는 원자적 쓰기 + 백업 필수**: 편집 흐름 = "YAML syntax → `SourcesConfig.model_validate` 통과 시에만 저장". 저장 전 `data/backups/sources/YYYYMMDD_HHMMSS.yaml` 타임스탬프 백업. 쓰기는 `NamedTemporaryFile` + `os.replace` 원자적; (a) EXDEV (tmp 와 target 이 다른 mount), (b) EACCES / EPERM (`/run/config/` 부모 dir 가 비쓰기 tmpfs) 두 fallback 은 `_overwrite_with_fsync`. 편집 대상은 bind mount 원본 — 다음 수집부터 반영.
- **Dockerfile 에 alembic 산출물 포함 필수**: `alembic.ini` 와 `alembic/` 가 이미지에 없으면 entrypoint 의 `alembic upgrade head` 가 "No 'script_location' key found" 로 실패. 빌드 시 반드시 COPY. 실제 운영에서는 `./alembic` · `./alembic.ini` 바인드 마운트가 COPY 보다 우선 적용되므로 신규 migration 추가 시 이미지 재빌드 불필요 — 이미지 내 COPY 는 마운트 누락 시 standalone 폴백 역할만 한다.
- **웹 수집 제어는 docker-in-docker 대신 호스트 docker.sock 원격 조작**: dind 는 이미지·권한·볼륨 재설계 부담이 커서, app 컨테이너에 `/var/run/docker.sock` 마운트하고 docker CLI + compose v2 플러그인을 설치해 `docker compose` 를 호출. 호스트 dockerd 가 실제 컨테이너를 관리하므로 scraper 의 바인드가 기존 CLI 경로와 동일하게 적용. 대가: docker.sock 마운트 = 사실상 호스트 root 권한 부여 — FastAPI 외부 노출 금지 전제가 더 강하게 요구된다. 설계 근거는 `docs/scrape_control_design.md §5`.
- **compose v2 필수 (v1 금지)**: v2 가 SIGTERM 을 관리 컨테이너 PID 1 로 공식 전파한다. v1 `docker-compose` 는 전파 동작이 검증되지 않아 "중단 → 공고 마무리 후 cancelled" 흐름이 깨질 수 있다. Dockerfile 이 compose v2 바이너리를 `/usr/libexec/docker/cli-plugins` 에 직접 배치해 `docker compose` 서브커맨드 형태로만 호출.
- **subprocess 는 새 프로세스 그룹 + 파일 redirect**: 웹·스케줄러 → scraper subprocess 기동 시 `start_new_session=True` (`os.setsid`) 로 새 프로세스 그룹. 중단은 `os.killpg(pgid, SIGTERM)` 로 그룹 전체 전파 — compose v2 가 컨테이너 PID 1 로 릴레이. detach 금지. stdout / stderr 는 PIPE 가 아닌 `data/logs/scrape_runs/{run_id}.log` **파일 redirect** (PIPE 버퍼 가득 차면 자식이 write 에서 블로킹). Windows 호환 포기 (`os.setsid` POSIX 전용).
- **ScrapeRun 을 세 경로 공통 lock 으로 사용**: 웹(`trigger='manual'`) / 스케줄러(`trigger='scheduled'`) / CLI(`trigger='cli'`) 모두 `create_scrape_run` 이 단일 트랜잭션에서 SELECT(running 검사)→INSERT. running row 가 곧 "실행 중 flag" — 별도 파일 락·메모리 뮤텍스 불필요. 웹 startup 의 `cleanup_stale_running_runs()` 가 pid 없는·사라진 running row 를 failed 로 정리해 lock 자동 해제. APScheduler `start_scheduler()` 는 cleanup 이후에 호출.
- **`SCRAPE_RUN_ID` 환경변수로 웹→CLI subprocess ScrapeRun 소유권 이전**: 웹·스케줄러의 `start_scrape_run` 이 `create_scrape_run` 으로 running row 를 먼저 INSERT 한 뒤 subprocess 기동. 기존 CLI 가 그대로 running 검사를 하면 자기 row 를 '진행 중인 다른 수집' 으로 오판해 exit(2) 하는 자기참조 버그가 있었다. `SCRAPE_RUN_ID` 환경변수로 id 를 주입하면 CLI 는 새 row 를 만들지 않고 해당 row 를 이어받아 finalize 까지 수행. 미주입 / 파싱 실패 시 직접 실행 경로 (자체 create_scrape_run, trigger='cli') 로 동작해 하위 호환 유지.
- **SIGTERM 처리는 atomic 경계 보존**: CLI 가 SIGTERM 을 받으면 handler 는 모듈 레벨 `_cancel_flag` 만 세우고, 실제 break 는 공고 / 소스 루프의 시작 지점에서만. 현재 공고 (첨부 다운로드 포함) 는 끝까지 마무리 — "UPSERT + 변경 감지 + 사용자 라벨링 리셋은 같은 트랜잭션" 원칙을 중단 경로에서도 유지하기 위함. 종료 status = `cancelled`.
- **`SCRAPE_ACTIVE_SOURCES` 환경변수로 per-run 소스 override**: 웹 UI "지금 시작 (소스 선택)" 은 `sources.yaml` 을 in-place 수정하면 entrypoint per-run 복사본 격리가 깨진다. 파일 대신 `SCRAPE_ACTIVE_SOURCES` (쉼표 연결, 예 `IRIS,NTIS`) 환경변수로 주입. CLI 가 로드 후 우선 적용; 미설정·빈 값이면 `sources.yaml` 의 `active_sources` 그대로.
- **컨테이너를 호스트 UID/GID 로 실행**: `docker-compose.yml` 두 서비스에 `user: "${HOST_UID}:${HOST_GID}"`. `.env` 명시 설정 필수 (자동 fallback 없음). `./data/` 하위 모든 런타임 생성 파일이 호스트 유저 소유가 되어 `sudo` 없이 관리 가능. 기존 root-owned 파일은 `sudo chown -R "$(id -u):$(id -g)" ./data/` 로 일회성 복구.
- **컨테이너 `/etc/passwd` 등록은 빌드 시 `useradd` 로 처리, entrypoint 동적 패치 불가**: 비특권 UID 로 기동 중인 컨테이너에서는 root 없이 `/etc/passwd` 수정이 불가하다. Dockerfile Layer 5a 에서 `ARG HOST_UID/HOST_GID` 를 받아 idempotent `useradd` (`-M -N`). UID 변경 후 `docker compose build` 재실행 없이는 "I have no name!" 재발. entrypoint 는 감지 시 경고만 출력하고 기동을 계속한다.
- **`HOST_DOCKER_GID`·`HOST_PROJECT_DIR` `.env.example` 기본값 제거 → 명시 입력 강제**: 호스트마다 docker 그룹 GID 가 다르므로 주석 처리된 999 같은 기본값을 빈 값으로 변경. README 초기화 절차에 `getent group docker | cut -d: -f3` (docker GID) 와 `pwd` (프로젝트 루트) echo 명령을 두어 신규 장비에서도 올바른 값이 자동으로 `.env` 에 기록되도록 한다.
- **`HOST_PROJECT_DIR` 는 `env_file: - .env` 단일 출처 — `environment` 블록 항목 제거 (00134)**: docker compose 에서 `environment` 는 `env_file` 보다 우선순위가 높다. `environment` 블록에 `HOST_PROJECT_DIR=${HOST_PROJECT_DIR:-}` 를 두면 outer compose 의 보간 컨텍스트(호출 쉘 + 탐색된 `.env`)에서 값을 찾지 못할 때 빈 문자열로 평가되고, 그 빈 값이 `env_file` 의 정상값을 덮어써 컨테이너의 `HOST_PROJECT_DIR` 가 `''` 가 된다 — 호출 CWD / compose 버전에 따라 보간 성공 여부가 갈리므로 "간헐적" 으로 보였다. 수정: `environment` 에서 `HOST_PROJECT_DIR` 항목을 제거하고 `env_file` 의 리터럴 읽기를 유일한 주입 경로로 확정. `COMPOSE_PROJECT_NAME` 은 `.env` 에 없을 수 있고 기본값(`iris-agent`)이 있어 `${VAR:-iris-agent}` 형태를 유지한다.
- **`as_utc` 헬퍼는 공용 (auth → 전역 승격)**: `_as_utc` 로 `app/auth/service.py` 에만 있던 helper 를 `app/db/models.py` 로 이동해 `scrape_runs` 비교 등에서 재사용. SQLite `DateTime(timezone=True)` 의 tz 손실은 프로젝트 공통 이슈라 컬럼 비교가 있는 모든 경로에서 동일 패턴이 필요.
- **stdlib→loguru 브리지 필수**: uvicorn 은 FastAPI 라우트의 미처리 예외를 stdlib `logging` 의 `uvicorn.error` 채널로 기록. loguru 와 stdlib 는 별도 시스템이라 브리지 (`InterceptHandler`) 없이는 stack trace 가 docker logs 에 전혀 나타나지 않는다 (LOG_LEVEL=DEBUG 에도 로그가 비어 보이는 근본 원인). 해결: `InterceptHandler` 를 root logger 에 부착 + uvicorn 로거 핸들러 명시적 교체.
- **Dockerfile 레이어 변경 빈도 오름차순 고정**: 소스 코드 변경 시마다 chromium (>200MB) · apt (~140MB) 를 다시 내려받는 문제 해결. 레이어 순서를 "거의 변경 없음 → 버전 고정된 무거운 도구 → 의존성 → 소스" 순으로 재편. playwright 버전은 Dockerfile `ARG PLAYWRIGHT_VERSION` 으로 핀해 동일 버전에서 Layer2 캐시 재사용. Dockerfile ARG (운영 핀) 와 pyproject 의 `playwright>=1.44,<2.0` (호환 범위) 를 두 곳에 의도적으로 분리.

### 이메일 발송 (`app/email/`)

- **Transport 는 Strategy 패턴 (ABC + 구현체 분리)**: 현재 구현체는 M365 OAuth SMTP 단 하나이나, 향후 Basic Auth SMTP 등 추가 시 factory 분기만 확장하면 되도록 설계. `send_with_retry` 는 구체 transport 타입을 모른다.
- **매 `send()` 마다 msal 인스턴스 새로 생성**: `ConfidentialClientApplication` 의 인메모리 토큰 캐시는 재사용을 고려했으나, 자격증명 갱신이 즉시 반영되도록 캐시 포기를 선택. transport 인스턴스 자체는 재사용해도 무방하나 캐시 효과는 없다.
- **재시도 backoff 는 단순 2초 flat, exponential 금지**: 메일 발송 실패는 대부분 일시적 네트워크/SMTP 서버 과부하. 운영 단순성을 위해 exponential backoff 를 도입하지 않음. `max_retry_count` 는 `SystemSetting["email.max_retry_count"]` → 호출자가 읽어 sender 에 전달 (sender 자체는 읽지 않음).
- **`send_with_retry` 가 `session.commit()` 직접 수행**: `EmailSendRun` 의 attempt_count·error_message 갱신을 "재시도 도중 process crash 시에도 이력 보존"하기 위해 sender 가 commit 책임을 진다. 호출자(admin API)는 이 점을 인지하고 별도 commit 을 하지 않는다.
- **`EmailSendRun.error_message` 는 마지막 시도의 예외만 보관**: 중간 시도의 에러는 로그로 기록되나 row 에는 남기지 않는다. 최종 성공 시 `error_message = NULL` 로 클리어.
- **client_secret 은 평문 DB 저장 (암호화 미구현)**: `system_settings` 테이블에 평문 저장. DB-level 암호화는 Phase A-1 범위 밖이며 향후 별도 작업 예정 (`docs/phase_a1_design_note.md §10`).
- **SMTP 호스트/포트·OAuth scope 는 코드 상수**: `smtp.office365.com:587` · `https://outlook.office365.com/.default` 는 옵션 B(M365) 의 정의 자체에 묶인 고정 값이라 `SystemSetting` 화하지 않고 `m365_oauth.py` 내부 상수로 박음.
- **`EmailForwardLog` 메시지 본문 미저장**: 개인정보·DB 크기 고려로 포워딩 메시지 본문은 저장하지 않고 `has_additional_message` boolean 만 기록. 제목·수신자 목록·성공/실패 카운트는 메타데이터로 보존.
- **`EmailForwardLog.status` 는 default 없음**: 포워딩 완료 후에야 결과가 확정되므로 ORM 레벨 default 를 두지 않고 Part 2 application 코드가 INSERT 시 명시적으로 채우도록 강제 (의도된 제약).
- **`email.send_enabled` 기본값 off — 최초 기동 시 의도치 않은 발송 방지**: `DEFAULT_EMAIL_SEND_ENABLED = False` 로 설정해 row 가 없거나 빈 값이면 off 로 fallback. 포워딩·테스트 발송 경로 모두 `app/email/gate.py::is_email_sending_enabled()` 진입 직후에 확인 — `EmailForwardLog` / `EmailSendRun` row INSERT 전에 차단해 이력이 남지 않도록 한다. 비활성화 상태에서 발송 시도 시 HTTP 503 + 안내 메시지 반환.
- **`EmailForwardLog` relationship 은 단방향**: `CanonicalProject`, `User`, `Organization` 을 향한 `lazy="select"` 단방향 relationship 만 선언. 상대 모델에서 `email_forward_logs` collection 을 조회할 활용 시나리오가 Part 1 에는 없어 back_populates 추가로 불필요한 ORM surface 를 늘리지 않음 (`EmailSendRun.requested_by` 와 동일 패턴).
- **`EmailForwardLog` ↔ `EmailSendRun` 연결은 application 레벨 (FK 없음)**: `EmailSendRun.related_kind='forward'`, `related_id=EmailForwardLog.id` 조합으로 연결. DB FK constraint 는 추가하지 않고 application 코드(`forwarding.py`)가 INSERT 시 채움. 발송 이력 expand 조회(`/forward-logs/{id}/sends`)는 이 두 컬럼으로 join. Phase A-1 은 `EmailForwardLog` 독립 테이블로 완결하고, Phase A-2 Part 2(00109)가 `related_kind`/`related_id` 연결을 완성.

### 공고 포워딩 (`app/email/forwarding.py`, Phase A-2 Part 2)

- **수신자 1명씩 개별 발송 (BCC 금지)**: 동보 메일(수신자 전체를 To/BCC 에 묶어 1회 발송) 대신 수신자마다 별도 `EmailMessage` 를 생성·발송. 프라이버시(수신자 목록 비노출) + per-recipient 발송 성공/실패 추적 + `EmailSendRun` row 개별 생성이 이유. 성능 희생이 있으나 수신자 최대 50명 상한으로 허용 범위.
- **`app.public_base_url` SystemSetting 으로 메일 본문 URL 관리**: 메일 본문에 공고 상세 URL(`{base_url}/announcements/{id}`)을 삽입할 때, 운영 환경마다 다른 서버 주소(예: `http://team-server.lan:8000`)를 `SystemSetting["app.public_base_url"]`로 관리. row 없으면 `http://localhost:8000` fallback (seed migration 없음). 관리자 이메일 설정 페이지 「시스템 접근 주소」 필드에서 변경 가능(00111). 입력 값은 http:// 또는 https:// 스킴 필수 — Pydantic validator 가 PUT 요청 시 422 로 거부.
- **포워딩 트랜잭션 3단계**: (1) `EmailForwardLog` 선(先) commit으로 "발송 시도 기록" 확보 → (2) 수신자별 `send_with_retry` 루프(각각 session.commit) → (3) 집계 후 최종 status/completed_at commit. `send_with_retry`가 호출자 session을 직접 commit하므로 미커밋 변경을 최소화하는 3단계 구조.
- **포워딩 canonical 단위, 발송은 최신 announcement 1건 기준**: 모달이 canonical_id를 받아 is_current=True·최신 버전 announcement를 선택해 메일 본문(제목·기관·마감일·링크)을 구성. 동일 과제의 여러 소스·버전 중 "대표 1건"을 자동 선택.
- **메일 본문에 발신자 정보 표 삽입**: 수신자가 누가 보냈는지 알 수 있도록 사용자 ID(필수)·발신 조직명(없으면 공란)·이메일(없으면 공란) 3행 표를 공고 메타 박스 바로 아래에 삽입. 발신 조직은 사용자가 선택한 단일 조직만 표시 (조직 미선택 시 공란). `message_builder` 는 `sender_organizations: list[Organization]` 시그니처를 유지하며, 호출자(`forwarding.py`)가 `[sender_organization]` 단건 또는 `[]` 를 전달.
- **목록 페이지 전달 버튼**: `/announcement` 목록 테이블에 "전달" 컬럼을 추가하고 ✉️ 버튼을 각 행에 노출. 로그인 + canonical_id 있을 때만 활성화 — 클릭 시 상세 페이지의 '메일로 보내기' 모달(`forward-open-btn` 이벤트)을 동일하게 열어 목록에서 바로 포워딩 가능. 비로그인 또는 canonical 없음이면 `disabled` 상태로 표시. colspan 비로그인 7→8, 로그인 8→9 로 증가.
- **포워딩 비동기화 + polling 방식 프로그레스 바 (00120)**: `POST /forward` 가 즉시 반환하고 발송 루프를 `BackgroundTasks` 로 위임해 수신자가 많아도 요청이 블로킹되지 않는다. 진행 현황은 SSE / WebSocket 대신 단순 polling(`GET /forward-logs/{id}`)으로 구현 — 구현 복잡도가 낮고 로컬 환경(단일 서버, 소수 사용자)에서 polling 부하가 무시 가능하기 때문. 프론트엔드 모달에 프로그레스 바 + "N명 중 M명 완료 / 실패 K건" 실시간 표시. 발송 중 상태는 `completed_at IS NULL` → `status:'in_progress'` 가상 값으로 API 에서 노출.

### 단체 Daily Report (`app/email/daily_report.py`, Phase A-3)

- **누적 구간 = `(last_sent_at, now]` — `SystemSetting` 이 single source of truth**: 마지막 발송 시각(`email.daily_report.last_sent_at`, ISO-8601 UTC) 이후 ~ 지금까지의 `scrape_snapshots` 만 누적 머지한다. `EmailDailyReportRun` 이력 테이블이 아니라 `SystemSetting` 한 곳만 참조 — cron 패턴이 어떻든(평일만, 주말 건너뜀 등) 자동으로 정확한 누적 구간이 계산된다. 시간 비교 컬럼은 `scrape_snapshots.created_at`.
- **첫 발송 fallback 7일**: `last_sent_at` 이 NULL·빈 값·파싱 불가이면 모두 "첫 발송 전" 으로 간주하고 직전 `fallback_days=7` 일치 snapshot 을 대상으로 한다. fallback_days 는 모듈 상수 1곳에 격리해 운영 데이터를 보고 1줄로 조정 가능.
- **snapshot 0건이어도 항상 발송**: 구간 내 snapshot 변화가 없어도 `SKIPPED` 로 빠지지 않고 빈 본문(5종 카테고리 모두 0건)으로 발송을 진행한다. `EmailDailyReportStatus.SKIPPED` enum 값은 과거 이력 row 호환을 위해 코드에서 제거하지 않는다.
- **`manual_test` 는 최근 24시간 고정 구간**: `TRIGGER_MANUAL_TEST` 발송은 `last_sent_at` SystemSetting 과 무관하게 `compute_aggregation_window(fixed_lookback_hours=TEST_SEND_LOOKBACK_HOURS)` 로 항상 `(now-24h, now]` 구간을 사용한다. 모듈 상수 `TEST_SEND_LOOKBACK_HOURS = 24` 로 단일 관리.
- **`last_sent_at` 갱신 정책표**: `scheduled`/`manual_admin` 트리거는 `SUCCESS`·`PARTIAL` 일 때만 갱신, `FAILED` 는 유지(같은 구간 재시도). **`manual_test` 트리거는 모든 상태에서 갱신하지 않는다** — 테스트 발송이 정규 발송 구간을 망가뜨리지 않도록. 정책은 `frozenset` 2종(`_TRIGGERS_THAT_MAY_UPDATE_LAST_SENT_AT`/`_STATUSES_THAT_TRIGGER_LAST_SENT_AT_UPDATE`)으로 코드 상수화해 표에 없는 트리거가 추가돼도 안전 default 가 "갱신 안 함".
- **수신자 = `is_admin` 사용자 이메일 자동 수집**: 별도 `SystemSetting` 수신자 명단을 두지 않는다. `collect_admin_recipient_emails` 가 `is_admin=True AND email IS NOT NULL/'' AND email_subscribed=True` 인 사용자만 수집(중복 제거 + case-insensitive 정렬). 이메일 미설정·수신거부 admin 은 자동 제외되며 admin UI 카드에 경고로 노출된다. "당분간 admin 검증" 단계용 — Phase A-4 의 EmailSubscription 으로 확장 예정.
- **`email.send_enabled` 게이트는 Daily Report 에도 적용**: 포워딩과 동일하게 `is_email_sending_enabled()` 가 off 면 자동 잡·테스트 발송·지금 발송 모두 차단(503) + `EmailDailyReportRun.status='failed'` row 는 남긴다. A-1 의 `/admin/email/test-send`(게이트 제외)와 달리 Daily Report 의 test-send 는 본 발송 동작을 그대로 검증하는 게 목적이라 게이트를 통과해야 한다.
- **트랜잭션 단계는 포워딩 패턴 미러**: (1) `EmailDailyReportRun` 선 commit(`in_progress`) → (2) 게이트 확인 → (3) trigger 기반 구간 계산(`TRIGGER_MANUAL_TEST` 면 24시간 고정, 그 외는 last_sent_at 기반) → (4) 본문 빌드 → (5) 수신자별 `send_with_retry` 루프(`EmailSendRun.related_kind='daily_report'`, `related_id=run_id`) → (6) 최종 status/`completed_at` commit → (7) 정책표 판정 후 `last_sent_at` 별도 commit.
- **APScheduler cron 잡은 KST 기준**: `BackgroundScheduler(timezone=KST)` 가 이미 KST 고정이라 cron 표현식(`0 9 * * 1-5` = 평일 09:00 KST)도 KST 로 해석 — UTC 변환 없음. 잡 ID `daily-report`(백업 잡 `backup-db` 패턴). `register_daily_report_cron_schedule` 가 `SystemSetting` 토글을 source of truth 로 등록/갱신/제거를 일치시키고(`enabled=False` 또는 빈 cron 이면 잡 제거), startup 시 `ensure_daily_report_cron_registered()` 가 자동 복원한다.
- **본문 레이아웃**: 5종 카테고리 섹션(각각 `border:1px solid #e0e0e0` 단일 셀 표로 감쌈) + 공고 행(`<table>/<td>` 열 레이아웃, 셀 padding 간격 보장) + 발송 메타 박스·footer. forward 빌더의 grayscale 패턴 재사용.

### UI / UX

- **bulk 요청 두 가지 모드 — `ids` vs `filter`**: 페이지 내 선택은 `ids` 모드(명시 목록), "현재 필터 결과 전체 M건 선택" 은 `filter` 모드(서버가 필터 파라미터로 announcement id 추출). `MAX_BULK_MARK` 상수로 한 번에 처리할 수 있는 상한 제한. 클라이언트가 필터 재구성을 서버에 위임해 페이지를 넘는 대량 선택을 지원.
- **배지 클릭 vs row 클릭 충돌은 `stopPropagation` 으로 해결**: 목록 row 전체가 상세 링크(`<a>`) 로 감싸져 있을 때 내부 배지 클릭이 상세 페이지로 이동하지 않도록 `event.stopPropagation()` + `preventDefault()`. 클릭 이벤트가 있는 인라인 위젯 추가 시 동일 패턴.
- **테이블 셀 overflow 클리핑을 벗어나는 툴팁은 `position: fixed` + JS `getBoundingClientRect()` 패턴**: CSS `:hover` 트리거 대신 JS `mouseenter/focusin` 이벤트로 전환. `getBoundingClientRect()` 로 viewport 좌표를 계산해 `fixed` 레이어에 배치 — 부모 요소의 `overflow: hidden` 클리핑을 탈출. 위 공간 부족(8px 미만) 시 아래로 자동 반전 (`rj-tooltip--below` 클래스 토글), 좌우 viewport 경계 8px 여백 클램핑. 로그인 여부 무관하게 IIFE 최상단에서 `initTooltips()` 를 호출해 비로그인 사용자에게도 카운터 툴팁이 표시된다.
- **관련성 모달 — '판정 취소' 버튼 제거, 본인 판정 목록 X 삭제 패턴 채택**: 모달 내 단일 "판정 취소" 버튼 대신, 하단에 `GET /mine` 으로 불러온 본인 판정 목록을 렌더링하고 행마다 X 버튼으로 `DELETE /canonical/{id}/relevance` 를 호출하는 방식으로 통합. X 버튼 클릭 시 `window.confirm()` 확인 후 삭제 진행 (커스텀 모달 중첩 없이 OS 네이티브 UI 사용). 사유가 있는 항목은 판정 배지 아래에 사유 텍스트를 추가 표시 (`flex-direction: column` + `.rj-modal__mine-reason` 서브 행; 사유 없으면 서브 행 미렌더링). 판정 삭제 후 목록 자동 재갱신 (페이지 새로고침 없음). 판정이 없으면 "등록된 판정이 없습니다" 안내 표시. 판정 입력 영역은 조직 판정 전용 — 개인 판정 입력(라디오·개인/조직 구분) 제거됨.
- **sources.yaml credentials 슬롯 (optional)**: 로그인이 필요한 소스를 위한 `credentials.username` / `credentials.password` 필드를 `SourceConfig` 에 예약. NTIS 는 게스트 수집이 가능해 현재 미사용. 실제 값은 `.env` 로만 주입하고 yaml 에는 키 이름만.
